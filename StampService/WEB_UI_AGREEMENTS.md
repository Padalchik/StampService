# StampService Web UI: рабочие договорённости

Актуально на 2026-06-15, Europe/Moscow.

Этот файл фиксирует договорённости по разработке полноценного web-интерфейса StampService, чтобы не переобсуждать базовые решения в новых задачах.

## Цель

Разработать полноценное React web-приложение для StampService.

В конечном состоянии web-приложение должно иметь функционал, аналогичный текущему Telegram-боту. При этом бизнес-логика не должна переноситься во frontend или API controllers: правила остаются в Domain/Application, а web/API остаются thin UI/API layer.

## Первая целевая аудитория

Первый web-сценарий покрывает:

- клиента;
- сотрудника бренда;
- владельца бренда.

Глобальная системная админка не является первым приоритетом, если она явно не запрошена отдельной задачей.

## Первый этап реализации

Первым этапом сделать только авторизацию по телефону.

Авторизация должна следовать существующим правилам приложения:

- вход по телефону через OTP;
- первичный вход/авторизация пользователя происходит только через OTP по телефону; сам аккаунт также может быть заранее создан бизнес-сценарием начисления по телефону;
- активный пользовательский аккаунт должен иметь активную `Phone` identity;
- OTP создаётся существующими Application-сценариями;
- доставка кода идёт через текущую реализацию `IPhoneAuthCodeSender`;
- обычная доставка кода в dev/admin-сценарии идет администратору в Telegram;
- SMS-доставка клиенту доступна отдельным явным действием пользователя и зависит от системной настройки;
- подтверждение первичного телефонного входа должно опираться на актуальный активный OTP для нормализованного номера телефона, а не на технический id UI-сценария;
- после подтверждения кода frontend получает JWT;
- frontend не показывает пользователю OTP-коды, debug-данные, internal ids и служебные поля.

На экране входа есть два разных действия:

- `Получить код` - базовый сценарий, отправляет OTP администратору в Telegram и не отправляет SMS клиенту;
- `Получить код по СМС` - явный SMS-сценарий, передает в API запрос с `sendSms=true`.

Кнопка `Получить код по СМС` должна быть недоступна, если публичная системная настройка SMS-кодов выключена или еще загружается. Frontend проверяет настройку для UX, но не является источником безопасности: backend/Application все равно обязан отказать SMS-сценарию, если SMS выключены. Пользовательский текст ошибки для такого отказа - обычная понятная формулировка вроде `Отправка кода по SMS сейчас выключена.`, без технических кодов.

Telegram не является первичным способом создания аккаунта в web. Web может предлагать первичную привязку Telegram только из уже авторизованной телефонной сессии, если у пользователя еще нет активной Telegram identity.

Если при реализации возникнет неясность по правилам авторизации, нужно остановиться и уточнить правило у владельца проекта.

## Локальный запуск web-стенда

Для ручного тестирования web-приложения используется VS Code launch configuration:

- `App: полный запуск в браузере`

Эта конфигурация одной кнопкой запускает:

- `StampService.API`;
- `StampService.TelegramBot`;
- React/Vite frontend;
- браузер на странице входа web-приложения.

PostgreSQL не должен подниматься этой конфигурацией. Владелец проекта запускает PostgreSQL самостоятельно.

Для текущей телефонной авторизации важно, что OTP-код доставляется администратору в Telegram. Поэтому для web-входа должны быть выполнены условия:

- `StampService.API` запущен;
- `StampService.TelegramBot` запущен для обычной Telegram UX;
- Telegram bot token доступен конфигурации API, потому что отправку OTP из web-сценария выполняет API через `IPhoneAuthCodeSender`;
- `Admin:TelegramUserIds` настроен и содержит Telegram id администратора.

Для проверки SMS-доставки дополнительно нужны `SmsAero:Login` и `SmsAero:ApiKey` в конфигурации API host, обычно через .NET user-secrets в dev или переменные окружения на стенде. Системная настройка включения SMS-кодов управляется из web-админки и хранится в БД; `SmsAero:SendAuthCodes` в конфигурации не является рабочим переключателем после создания БД-настройки, а используется только как стартовое значение.

## Формат frontend

Web UI должен быть полноценным frontend-проектом на React.

Ориентир:

- React;
- TypeScript;
- нормальная структура приложения: роутинг, API-клиент, auth state, layout, страницы, общие компоненты;
- не развивать static `wwwroot`-страницы как web-поверхность.

Старые static/prototype web surfaces в `src/StampService.API/wwwroot` удалены из целевой архитектуры. Маршруты `/app`, `/phone-auth-test` и `/design-variants` не являются рабочими web-сценариями и не должны восстанавливаться без отдельной причины. Новые web-задачи развиваются в `src/StampService.Web` как React/TypeScript app поверх typed API clients.

## Визуальный стиль

На первом этапе приоритет - функциональность, ясность и устойчивость интерфейса, а не визуальная полировка.

Правила:

- интерфейс только на русском языке;
- тексты пользовательские, без технических формулировок;
- в React Web UI пользовательский термин для доменных метрик - `Штампы`; внутренние `Metric*`, `metrics`, `Metrics` и API endpoints не переименовываются в рамках UI-текстовых задач;
- в web-навигации не использовать emoji: web должен быть спокойным рабочим интерфейсом, даже если в Telegram для тех же сценариев используются emoji;
- повторяемые названия разделов и навигации хранить в одном локальном frontend-источнике, а не дублировать строками по компонентам;
- не показывать пользователю ids, debug, internal codes и служебные поля;
- disabled/недоступные возможности бренда не должны протекать в UI;
- дизайн должен быть спокойным рабочим интерфейсом, пригодным для клиента, сотрудника и владельца бренда.

## Мобильная адаптация

React Web UI должен быть пригоден для мобильного формата без горизонтального съезда элементов. Текущая адаптация выполнена в `src/StampService.Web/src/styles.css` через общие responsive rules, а не через изменение бизнес-компонентов или API.

Архитектурный принцип: компоненты остаются теми же, а layout меняется общим shell/CSS-слоем. На desktop это рабочий dashboard/workspace с боковой навигацией и многоколоночными блоками. На mobile те же сценарии складываются в одну колонку, действия становятся полноширинными там, где это нужно для читаемости, а основная навигация доступна снизу.

Текущие контрольные точки:

- `max-width: 760px` - основной mobile breakpoint: `app-shell`, workspace, карточки, формы, operation grids, brand/admin/profile/wallet blocks переходят в mobile layout; desktop/sidebar-навигация скрывается, а вместо нее используется fixed bottom navigation;
- `max-width: 420px` - дополнительный узкий breakpoint: меньше отступы и более компактные элементы bottom navigation;
- bottom navigation рендерится из того же общего списка доступных navigation items, что и desktop sidebar; отдельной mobile role/access matrix быть не должно;
- mobile labels должны оставаться короткими: `Кошелёк`, `Работа` при одном рабочем бренде, `Бренды` при нескольких, `Админка`, `Аккаунт`;
- `Админка` не прячется внутрь аккаунта: если `GET /api/admin/access` разрешил доступ, это отдельный пункт desktop и mobile навигации;
- workspace на mobile должен иметь нижний отступ под fixed bottom navigation с учетом `env(safe-area-inset-bottom)`, чтобы навигация не перекрывала последнюю карточку или кнопку;
- классы вроде `.workspace__header`, `.section-heading--split`, `.wallet-code-panel`/`.wallet-code-card`, `.brand-list-item`, `.operation-option`, `.metric-management-row__summary`, `.staff-row` на mobile складываются вертикально;
- desktop grid-классы должны иметь mobile override, если они могут оказаться шире экрана;
- админский список брендов использует собственные `.admin-*` классы и следует паттерну `BrandSelector`: компактная строка, avatar первой буквы, пользовательские данные и действие справа;
- длинные названия, статусы, кнопки и коды списания должны проектироваться так, чтобы не расширять viewport: использовать существующие responsive-классы, `min-width: 0`, перенос текста и полноширинные действия на mobile.
- segmented controls должны оставаться однострочными для коротких рабочих подписей вроде `Сотрудники`; на mobile можно уменьшать отступы/шрифт и распределять ширину flex-ом, чтобы вкладки не ломались на две строки.

Новые web-экраны нужно сразу проверять в мобильной модели: не добавлять фиксированные desktop-колонки без mobile override, не рассчитывать на то, что `overflow-x` скроет проблему, и не переносить бизнес-решения во frontend ради удобства верстки.

## Авторизация и сессия

Используется JWT, потому что backend уже поддерживает JWT.

Базовая модель сессии: пользователь входит по телефону, получает JWT для `User.Id`, у которого есть активная `Phone` identity. Telegram-login не должен создавать новый `User` без подтвержденного телефона; если такой endpoint сохраняется для совместимости, web не должен использовать его как первичный вход.

То же правило применяется к первичному входу через Telegram-бот: если Telegram identity еще не привязана, бот ведет пользователя через подтверждение телефона и затем привязывает Telegram к найденному или созданному телефонному аккаунту. Если аккаунт уже создан в web по телефону, вход в Telegram по этому же телефону должен привязать текущий Telegram id к тому же `User`. Такой phone-first onboarding концептуально равен телефонному входу, поэтому подтверждение OTP не должно зависеть от `AuthCodeId` из Telegram session. `AuthCodeId` допустим только для stateful-сценариев профиля, где identity еще не привязана.

Штатная смена телефона в web/Telegram UX реализована отдельным flow: пользователь уже находится в подтвержденной сессии своего `User.Id`, новый номер подтверждается OTP, старая `Phone` identity soft-delete, новая `Phone` identity добавляется к тому же `User.Id`. Перенос phone identity между пользователями запрещен. Штатной перепривязки Telegram нет; миграция legacy Telegram-only аккаунтов допускается только как отдельный явно согласованный flow.

Для web это означает, что любые сценарии входа, профиля, клиентских операций, staff/admin lookup и фильтров по телефону должны опираться только на активные identity. Soft-deleted phone identity - это история привязки, а не допустимый способ входа или поиска пользователя. Frontend не должен пытаться "помочь" пользователю старым номером: UI собирает текущий ввод, backend/Application нормализует его и ищет активную `Phone` identity.

Так как владелец проекта не хочет самостоятельно принимать слабые security-решения, реализация должна ориентироваться на best practices:

- изолировать работу с токеном в отдельном auth/API слое frontend;
- не размазывать JWT-логику по компонентам;
- централизованно обрабатывать `401 Unauthorized`;
- иметь явный logout; в текущем web UX выход находится в разделе `Аккаунт` как действие `Выйти из аккаунта`, а не в общем header приложения;
- не логировать токен и чувствительные данные;
- хранение токена выбрать осознанно при реализации и зафиксировать причину в задаче/ответе.

Если потребуется изменить backend-авторизацию, например перейти на httpOnly cookie/refresh-token flow, это нужно обсуждать как отдельное архитектурное решение.

## Телефонные номера

Единое правило для телефонных номеров:

- пользовательский ввод должен приводиться к международному формату с `+`, например `+79991234567`;
- базовый синтаксис соответствует E.164: `+`, далее цифры, первая цифра не `0`, максимум 15 цифр;
- для номеров с кодом `+7` дополнительно требуется ровно 11 цифр вместе с кодом страны, например `+79991234567`;
- разрешены цифры и обычные разделители ввода: пробел, дефис, круглые скобки;
- буквы, несколько `+`, номер без `+`, слишком короткий или слишком длинный номер считаются ошибкой;
- внутри системы телефон хранится в нормализованном виде без разделителей, например `+79991234567`;
- backend остаётся авторитетным источником валидации;
- web и Telegram UI могут делать раннюю UX-проверку, но не должны обходить Domain/Application validation;
- в web-полях телефона используется маска для российских номеров: поле закрепляет префикс `+7`, пользователь вводит локальные цифры, UI форматирует значение как `+7 (999) 123-45-67`, а перед отправкой нормализует его в `+79991234567`;
- если пользователь начинает ввод с `7` или `8`, web-маска воспринимает это как попытку ввести код страны и не добавляет эти цифры в локальную часть номера.

Общее правило C#-валидации находится в доменной модели телефона, а Application-сценарии используют его через нормализацию перед созданием OTP и phone identity.

## Роли и навигация

Desktop/tablet использует левый sidebar. Mobile (`max-width: 760px`) использует fixed bottom navigation как основной способ перехода между разделами; burger/скрытое меню не является базовой моделью навигации.

Ориентир навигации:

- клиентские разделы доступны пользователю как базовая часть приложения;
- если пользователь является сотрудником или владельцем бренда, показывать разделы рабочих брендов;
- действия в бренде показывать с учётом роли и feature toggles;
- недоступные действия лучше скрывать, как сейчас в Telegram UX, если нет отдельной причины показывать disabled-состояние;
- `AppShell` должен формировать единый список доступных navigation items из существующего состояния (`GET /api/brands/mine`, `GET /api/admin/access`) и затем рендерить его разными layout-ами: desktop sidebar и mobile bottom navigation;
- не дублировать проверки ролей/прав отдельно для mobile;
- основные web labels и короткие mobile labels хранятся в `src/StampService.Web/src/app/navigationLabels.ts`;
- `Настройки аккаунта`/`Аккаунт` остаются отдельным пунктом навигации; `Админка`, если доступна, тоже отдельный пункт.

## Telegram parity и API

Целевое правило: web должен постепенно повторить экраны и функционал Telegram-бота.

Важная архитектурная оговорка: Telegram-бот часто вызывает Application use cases напрямую, а не HTTP endpoints. Поэтому факт, что сценарий уже работает в Telegram, означает, что бизнес-сценарий в Application, скорее всего, уже есть. Но это не гарантирует, что для него уже есть HTTP endpoint, пригодный для web frontend.

Telegram parity не означает буквальное совпадение подписей кнопок. UI-copy может отличаться по каналу: Telegram может использовать emoji, web - нет. При этом внутри каждого UI названия должны быть централизованы локально.

Для клиентских операций начисления web и Telegram следуют одной продуктовой модели: штампы (внутренне метрики) и монетки начисляются по номеру телефона клиента. Если клиент с такой активной `Phone` identity уже есть, начисление идет существующему `User`; если клиента еще нет, Application создает полноценного `User` с `Phone` identity и сразу проводит начисление через ledger. 4-значный `CustomerCode` не должен быть вводом в новых UI-сценариях начисления; старые HTTP/Application ветки начисления по `CustomerCode` удалены. Новые телефонные аккаунты больше не получают `CustomerCode` при создании, поэтому web/API не должны рассчитывать на наличие такого поля даже как на legacy-display value.

Важно: `CustomerCode` больше не является частью актуальной доменной и EF-модели `User`. Для web это означает не только “не показывать поле”, но и “не проектировать новые DTO, формы, фильтры, ссылки или типы вокруг customerCode”. Внутри API/Application владелец данных - `User.Id`; внешний пользовательский ввод для клиентских операций - телефон или одноразовый код списания в зависимости от сценария.

Важная граница: решение “найти или создать клиента по телефону” живет в Application use cases (`IssueMetricByPhoneCommand`, `IssueCoinsByPhoneCommand`), а не в React, API controller или Telegram endpoint. UI-слои только собирают телефон, количество и комментарий, затем вызывают общий сценарий. Это нужно, чтобы web и Telegram не расходились в поведении.

Та же граница применяется к уведомлениям клиента о начислении и списании. Уведомление не должно жить в React, API controller или Telegram endpoint. Оно запускается внутри общих Application handlers через `ICustomerNotificationService`, поэтому web и Telegram получают одинаковое поведение: если у клиента есть активная Telegram identity и настроен bot token, после начисления штампов/монеток, списания штампов, ручного списания монеток или выдачи товара за монетки ему уходит сообщение об операции. Web/API остаются thin layer и не должны отдельно дергать Telegram-доставку.

В notification-тексте после операции должен быть блок `Доступно сейчас`: список наград текущего бренда, которые клиент может получить при текущих балансах. Это не только “новые” награды, достигнутые последней операцией; если награда была доступна раньше и остается доступной сейчас, она тоже показывается. Если ничего не доступно, уведомление явно говорит, что сейчас доступных наград нет.

Списание остается отдельной моделью: штампы, ручное списание монеток и выдача товаров за монетки выполняются по одноразовому коду списания клиента, а не по телефону.

Результаты операций с монетками не должны возвращать или показывать `CustomerCode`. Общий `CoinOperationResponse` для API/web/Telegram содержит данные операции, имя клиента, сумму и баланс; старый 4-значный код не является частью публичного контракта. В web brand workspace и Telegram после начисления/списания/выдачи товара показываются пользовательские данные операции, но не legacy-идентификатор.

Просмотр клиентских балансов и истории операций сотрудником использует телефон клиента как внешний идентификатор, но не создает клиента автоматически. В отличие от начисления, это read-only сценарий: Application нормализует телефон, ищет существующую активную `Phone` identity и возвращает понятный отказ, если клиент не найден.

Управление сотрудниками бренда следует тому же phone-first направлению, но с другой бизнес-границей: добавление сотрудника не создает нового пользователя. Сотрудник/owner вводит телефон будущего сотрудника, Application нормализует номер, ищет существующий `User` по активной `Phone` identity и добавляет `BrandMembership` с ролью `STAFF`. Если пользователя с таким телефоном нет, сценарий должен отказать. Это сделано через `AddBrandStaffByPhoneCommand`; Telegram staff-flow уже использует этот подход и больше не вводит/не показывает `CustomerCode` для сотрудников. Когда аналогичный web UI будет добавляться, он должен быть thin UI поверх этого же Application-сценария, а не принимать `CustomerCode` или внутренний `UserId` как пользовательский ввод.

Системные админские сценарии назначения владельца бренда тоже переведены на телефон. Создание бренда с владельцем и смена владельца используют существующего пользователя с активной `Phone` identity: `CreateBrandWithOwnerCommand` принимает телефон владельца, `ReassignBrandOwnerCommand` - телефон нового владельца. Auto-create пользователя в этих сценариях запрещен, потому что это управление доступом. Telegram admin-flow уже следует этому правилу; если появится web/admin UI, он должен принимать телефон, а не `CustomerCode` или внутренний `UserId`.

Демо-инструменты также не должны возвращаться к `CustomerCode`. Создание пользовательских демо-данных выполняется по телефону существующего аккаунта через `CreateUserDemoDataCommand`: Application ищет активную `Phone` identity и наполняет выбранный бренд демо-балансами/историей через ledger-сервисы. Auto-create пользователя в demo-flow не выполняется. Если такой инструмент появится в web, UI должен принимать телефон и бренд, а не внутренние ids или старый код пользователя.

При разработке web-сценариев нужно:

1. Сначала смотреть, как соответствующий сценарий реализован в Telegram-боте.
2. Находить Application command/query, который использует Telegram.
3. Проверять, есть ли уже HTTP endpoint для этого сценария.
4. Если endpoint есть - использовать его.
5. Если endpoint отсутствует - сообщить владельцу проекта и согласовать добавление thin API endpoint поверх существующего Application use case.

Нельзя дублировать бизнес-логику из Telegram-бота во frontend.

## Web: рабочие бренды и клиентские операции

Раздел `Рабочие бренды` реализован как первая web-поверхность для сотрудника/owner. Цель раздела - дать web-аналог Telegram-механизма работы с клиентами и базовых owner-операций бренда.

Архитектурная модель:

- frontend показывает список рабочих брендов текущего пользователя;
- открытие бренда загружает workspace: роль, permissions и feature toggles;
- UI скрывает действия, которые недоступны по роли или отключены настройками бренда;
- frontend не знает бизнес-правил балансов и не меняет их напрямую;
- API endpoints остаются тонким слоем поверх Application use cases;
- все операции с балансами проходят через существующие Application handlers и ledger-сервисы;
- управление доступами и настройками также остается в Application; React только собирает ввод, показывает состояние и вызывает HTTP API.

Навигация web должна скрывать недоступные разделы, как Telegram bot. `AppShell` не должен показывать `Рабочие бренды`, если у пользователя нет membership в брендах, и не должен показывать `Админку`, если `GET /api/admin/access` возвращает `false`. При одном рабочем бренде пункт называется `Работа` и сразу открывает workspace этого бренда; при нескольких брендах desktop показывает `Рабочие бренды`, а mobile - короткое `Бренды`. Это только UX-фильтр; прямые вызовы API всё равно проверяются Application handlers.

Текущие web-сценарии:

- выдача штампа по номеру телефона клиента;
- списание штампа по одноразовому коду списания клиента;
- просмотр балансов клиента по телефону, если workspace вернул `CanViewBalances`;
- начисление монеток по номеру телефона клиента;
- ручное списание монеток по одноразовому коду списания;
- выдача товара за монетки по одноразовому коду списания;
- управление штампами бренда: список, создание, редактирование;
- управление товарами за монетки: список, создание, редактирование, удаление/deactivate;
- управление сотрудниками бренда: список, добавление по телефону, удаление;
- настройки бренда: включение/выключение штампов, монеток и способов списания.

Web API/types для этих операций должны оставаться синхронизированы с phone-first моделью: DTO результатов не включают `customerCode`, а React не должен добавлять это поле в локальные типы или UI. Если нужен контекст клиента на экране результата, достаточно имени клиента и данных операции; идентификация клиента для новых операций идет через телефон или одноразовый код списания в зависимости от сценария.

Управление сотрудниками бренда в web реализовано как часть brand workspace. Архитектурное правило соответствует Telegram/Application-реализации: пользовательский ввод - телефон сотрудника; Application use case - `AddBrandStaffByPhoneCommand`; auto-create пользователя не выполняется; отображение сотрудников использует телефон и имя, но не `CustomerCode`. Внутренний `UserId` может использоваться только для действий с уже загруженным сотрудником, например удаления через `RemoveBrandStaffCommand`, и не должен становиться пользовательским вводом.

Управление штампами и товарами за монетки в web использует существующие Application use cases через thin API endpoints. Технически штампы остаются доменными метриками (`Metric*`), поэтому API и typed client продолжают использовать прежние имена. Frontend не принимает доменные решения об активности, доступах или валидации; он делает раннюю UX-проверку обязательных полей и положительных чисел, а backend/Application остаются источником истины.

Настройки бренда в web управляют только feature toggles: `IsMetricsEnabled`, `IsCoinsEnabled`, `IsCoinProductRedemptionEnabled`, `IsManualCoinRedemptionEnabled`. UI повторяет Telegram-правило: должен быть включен хотя бы один тип наград, а если монетки включены, нужен хотя бы один способ списания. Сохранение идет через `UpdateBrandRewardSettingsCommand`; после успешного ответа workspace обновляет локальное состояние, чтобы скрыть недоступные секции.

Важное отличие web от Telegram: Telegram flow хранит промежуточное состояние в bot session, а web делает обычные HTTP-запросы из React state. Поэтому для web добавлены endpoints, которые принимают web-friendly входные данные. Например, выдача штампа принимает номер телефона клиента и технически передает его в `IssueMetricByPhoneCommand`; Application сам нормализует телефон, находит или создает клиента и проводит начисление. Web не должен принимать или показывать внутренний `UserId` клиента.

Для web-форм начисления телефонная маска используется как UX-помощник, а не как источник бизнес-правил. Frontend отправляет нормализованный телефонный номер в thin endpoint, backend/Application повторно валидирует номер и выполняет общий issue-by-phone flow.

Ключевые API endpoints:

- `GET /api/brands/mine`;
- `GET /api/brands/{brandId}/workspace`;
- `GET /api/brands/{brandId}/metrics`;
- `POST /api/brands/{brandId}/metrics`;
- `PUT /api/metrics/{metricDefinitionId}`;
- `GET /api/brands/{brandId}/metrics/issue-options`;
- `GET /api/brands/{brandId}/metrics/redeem-options?redemptionCode=...`;
- `POST /api/metrics/{metricDefinitionId}/issue-by-phone` - web-friendly thin endpoint поверх `IssueMetricByPhoneCommand`;
- `POST /api/metrics/{metricDefinitionId}/redeem`;
- `POST /api/brands/{brandId}/coins/issue-by-phone` - web-friendly thin endpoint поверх `IssueCoinsByPhoneCommand`;
- `POST /api/brands/{brandId}/coins/redeem`;
- `GET /api/brands/{brandId}/coin-products`;
- `POST /api/brands/{brandId}/coin-products`;
- `PUT /api/coin-products/{productId}`;
- `DELETE /api/coin-products/{productId}`;
- `GET /api/brands/{brandId}/coin-products/purchase-options?redemptionCode=...`;
- `POST /api/brands/{brandId}/coin-products/{productId}/purchase`;
- `GET /api/brands/{brandId}/staff`;

- `DELETE /api/brands/{brandId}/staff/{staffUserId}`;

Старые endpoints без phone-first модели не являются частью web-контракта: `POST /api/brands/{brandId}/staff` и `POST /api/metrics/{metricDefinitionId}/issue` удалены. Web должен использовать `staff/by-phone` и `issue-by-phone`, а не пытаться передавать внутренний `UserId` клиента/сотрудника как пользовательский ввод.

Ключевые frontend места:

- `src/StampService.Web/src/brands/BrandWorkspacePage.tsx` - контейнер выбора бренда и загрузки workspace;

## MVP-порядок

Согласованный порядок развития web-приложения:

1. Скелет web app: React-проект, роутинг, auth guard, API client, layout, обработка ошибок.
2. Телефонный вход.
3. Профиль, первичная привязка телефона при необходимости и привязка Telegram.
4. Кошелёк клиента с кодом списания, наградами и историей.
5. Brand workspace для сотрудника/owner. Базовая клиентская работа добавлена: выдача/списание штампов, монеток и товаров через thin API + Application use cases.
6. Управление штампами, товарами, сотрудниками и настройками бренда добавлено в тот же workspace через thin API + Application use cases.

## Web: личный кабинет

Экран `Личный кабинет` в web должен соответствовать данным профиля Telegram-бота и использовать общие Application-сценарии.

Архитектурное правило личного кабинета: телефон является обязательной первичной identity активного аккаунта, Telegram - вторичная привязка. Сценарии профиля не должны создавать Telegram-only аккаунты. Уже активная phone identity меняется только через отдельный OTP-flow смены телефона; Telegram identity не перепривязывается штатным профилем.

Текущая реализация:

- frontend загружает профиль через `GET /api/users/me`;
- профиль показывает имя и статусы привязки телефона/Telegram; `CustomerCode` не является публичным идентификатором и не возвращается в DTO профиля;
- первичная привязка телефона использует тонкие endpoints `POST /api/users/me/phone/code` и `POST /api/users/me/phone/verify`, если у профиля еще нет активного телефона;
- смена телефона использует тонкие endpoints `POST /api/users/me/phone/change/code` и `POST /api/users/me/phone/change/verify`, если у профиля уже есть активный телефон;
- первичная привязка Telegram в web идёт через deep link, а не через ввод Telegram id или username;
- frontend запрашивает ссылку через `POST /api/users/me/telegram/link`;
- API возвращает ссылку вида `https://t.me/<bot_username>?start=<token>`;
- пользователь открывает Telegram по ссылке, TelegramBot получает `/start <token>` и привязывает реальный Telegram user id к текущему web-пользователю через Application-сценарий;
- для генерации ссылки API должен иметь `Telegram:BotUsername`, а для обработки deep link должен быть запущен `StampService.TelegramBot`;
- username Telegram может показываться как display value, но не используется как устойчивый ключ привязки.
- выход из текущей web-сессии находится в этом разделе как пользовательское действие `Выйти из аккаунта`; общий app header не должен дублировать logout.

Правило привязки identity:

- если пользователь входит по телефону, Application находит или создает единственный телефонный `User`;
- если пользователь привязывает Telegram из web, Telegram identity добавляется к текущему телефонному `User` только при отсутствии активной Telegram identity у этого `User`;
- если телефон или Telegram уже принадлежит другому `User`, обычный web-сценарий должен отказать с понятной причиной;
- если у текущего `User` уже есть активная Telegram identity, web не показывает действие перепривязки Telegram, а backend дополнительно отказывает при прямом вызове endpoint;
- автоматическое объединение аккаунтов и перенос `UserIdentity` между пользователями не являются базовым пользовательским сценарием новой модели, потому что новые Telegram-only аккаунты больше не должны создаваться;
- если legacy/migration flow все же нужен, перенос наград и балансов должен идти через Application/ledger-действия, а не прямым изменением балансов.

## Web: мой кошелёк

Экран `Мой кошелёк` в web должен соответствовать данным Telegram-бота.

Текущая реализация:

- frontend вызывает `POST /api/wallet/open`;
- API endpoint должен быть тонким и вызывать Application-сценарий;
- общий Application-сценарий для web и Telegram - `OpenUserWallet`;
- `OpenUserWallet` собирает данные через существующие use cases:
  - создаёт или возвращает актуальный код списания;
  - загружает overview балансов и доступных наград;
  - возвращает код списания, срок действия кода и бренды;
  - best-effort отмечает открытие кошелька для digest/уведомлений.

Кошелек клиента больше не показывает и не получает `CustomerCode`. В phone-first модели публичный клиентский идентификатор - телефонная identity, а для списания используется отдельный одноразовый `RedemptionCode`, который подтверждает конкретную операцию у сотрудника. Новый код списания создается Application-сценарием на 10 минут; frontend может получать срок из DTO, но не должен вычислять его сам.

Так как `RedemptionCode` одноразовый, web-кошелек должен обновлять отображаемый код без ожидания ручного перезахода пользователя в раздел. Если сотрудник списал штампы/монетки или выдал товар по открытому у клиента коду, старый код больше не должен оставаться единственным видимым кодом на экране клиента. Текущая архитектурная модель для web - легкий background refresh карточки кода через typed API client `getCurrentRedemptionCode` и endpoint `POST /api/users/me/redemption-code`; ручная кнопка обновления передает `forceRefreshCode=true`, а фоновое обновление не должно перезагружать весь кошелек и не должно использовать `OpenUserWallet` как polling endpoint.

Текущий основной web UX кошелька сделан как быстрый mobile-first клиентский экран, а не как экран истории или полной детализации бренда:

- заголовок раздела `Мой кошелёк` показывается без описательного текста под ним;
- сразу под заголовком находится sticky-карточка `Код для списания` с крупным кодом и icon-only кнопкой обновления `RefreshCw` с `aria-label="Обновить код"`;
- срок действия кода остается в DTO/API, но на главном web-экране не выводится, чтобы не конкурировать с кодом списания;
- секция `Мои награды` не обернута в отдельную общую карточку: карточками являются только отдельные бренды как группы наград;
- в заголовке секции не показывается счетчик брендов; пустое состояние формулируется как отсутствие наград, а не отсутствие брендов;
- карточка бренда показывает название, одну короткую muted meta-строку и открывается кликом по всей карточке без отдельной кнопки `Открыть`;
- на главной карточке бренда показываются только конкретные доступные награды в виде chip-badge, максимум 3 видимых чипа и `+N ещё`, если наград больше;
- список доступных наград формируется из `availableCoinProducts[].productName` и `availableMetrics[].metricName` только для элементов с `isAvailable`;
- длинные названия наград должны быть устойчивы на mobile через `min-width: 0`, `max-width: 100%` и перенос текста.

Правило: вспомогательная отметка открытия кошелька не должна ломать отображение кошелька. Если marker/digest state не сохранился, основной экран кошелька всё равно должен загрузиться, а проблема должна логироваться.

Ручное обновление кода списания использует endpoint текущего кода с `forceRefreshCode=true`; фоновая актуализация использует тот же endpoint без принудительной ротации.

React dev mode может вызывать effects повторно, поэтому initial-загрузка кошелька во frontend должна быть защищена от двойного запроса.

Детализация бренда в web-кошельке:

- из списка брендов пользователь нажимает `Открыть`;
- детализация открывается как отдельный экран внутри раздела `Мой кошелёк`, а не как блок ниже списка;
- на экране детализации есть кнопка `Назад`, возвращающая к списку брендов;
- пункт навигации `Мой кошелёк`/`Кошелёк` также всегда возвращает на главный экран кошелька со списком брендов, даже если пользователь уже находится в разделе `wallet`;
- данные детализации должны соответствовать Telegram-сценарию:
  - доступные товары за монетки;
  - доступные штампы;
  - статус `доступно` / `не хватает`;
  - последние операции по монетам и штампам;
- Web и Telegram используют единый Application-сценарий детализации бренда: `GetUserWalletBrandDetailsQuery`;
- Web получает детализацию через тонкий endpoint `GET /api/wallet/brands/{brandId}/details`;
- отдельные web/API контракты для rewards и history не являются текущей поверхностью Web UI: данные наград и последние операции собираются в одном details-сценарии;
- UI-слои остаются разными: Application возвращает presentation-friendly DTO без Telegram HTML, React JSX и CSS, а Web и Telegram только рендерят этот DTO в свой формат.

Актуальная web-структура детализации бренда:

- основной экран кошелька находится в `src/StampService.Web/src/wallet/WalletPage.tsx`;
- экран наград выбранного бренда находится в `src/StampService.Web/src/wallet/BrandWalletPage.tsx`;
- общая карточка одноразового кода списания находится в `src/StampService.Web/src/wallet/RedemptionCodeCard.tsx`;
- общий блок вкладок `Штампы` / `Монетки` / `История` находится в `src/StampService.Web/src/wallet/WalletBrandDetailsBlock.tsx`;
- `BrandWalletPage` состоит из hero-плашки бренда, плашки кода списания, отдельной плашки вкладок и отдельной области контента; локальная кнопка `Назад` на этом экране не используется;
- единая большая `surface-panel` вокруг всего экрана не используется;
- название бренда и meta показываются в отдельной белой плашке;
- вкладки работают как segmented control внутри экрана без перезагрузки страницы;
- порядок вкладок: `Штампы`, `Монетки`, `История`;
- `Штампы` показываются только если у бренда включены метрики, `Монетки` - только если включены монетки; скрытые пункты не должны оставлять пустое место в segmented control;
- `details.hintText` в этом web-экране не рендерится;
- вся визуальная настройка находится в `src/StampService.Web/src/styles.css` через `brand-details-*`, `brand-reward-*` и `brand-history-*` классы.

Правила отображения наград:

- вкладка `Монетки` показывает текущий баланс монеток клиента в бренде;
- товары за монетки рендерятся внутри вкладки `Монетки` из reward section `CoinProducts`, если выдача товаров за монетки включена и секция пришла из Application;
- если выдача товаров за монетки выключена, вкладка `Монетки` остается доступной и показывает только баланс;
- вкладка `Штампы` рендерит элементы из технической reward section `Metrics`;
- frontend не должен придумывать стоимость, требования или нехватку: он использует `progressText`, `statusText` и `isAvailable` из DTO;
- порядок наград в `Штампах` и товарах внутри `Монеток`: сначала полностью доступные, затем недоступные с начатым накоплением, затем недоступные без прогресса;
- список изначально ограничен несколькими элементами и раскрывается кнопкой `Показать ещё`/`Показать все`;
- `progressText` показывается и для доступных, и для недоступных элементов как вторичная строка;
- progress bar допустим только там, где готовый `progressText` безопасно интерпретируется как прогресс.

Правила отображения истории:

- вкладка `История` сначала показывает пользовательский выбор источника;
- источники группируются по операции, но UI показывает только человекочитаемые имена: `Монетки` или `sourceName`;
- выбранный источник открывается в bottom sheet;
- список операций внутри bottom sheet должен скроллиться независимо от заголовка панели;
- в операции показываются только пользовательские поля: `amountText`, `formatRuDateTime(createdAt)`, видимый `comment`;
- `actorUserId`, technical `sourceType`, internal ids и debug-поля не выводятся.

Application details-сценарий должен возвращать web-ready данные. Доступность продуктовых разделов кошелька frontend берет из `UserWalletBrandDetailsResponse` (`isMetricsEnabled`, `isCoinsEnabled`, `isCoinProductRedemptionEnabled`, `coinBalance`), а не выводит из наличия reward sections. Для штампов техническая секция `Metrics` включает все активные метрики бренда, включая метрики с нулевым прогрессом у пользователя. Если у пользователя еще нет `MetricBalance`, Application возвращает item с текущим значением `0`, а React только рендерит готовый DTO и пользовательский label `Штампы`.
Reward sections описывают конкретные блоки наград, а не сами продуктовые разделы. Если учет метрик выключен, `Metrics` не возвращается; если выдача товаров за монетки выключена, `CoinProducts` не возвращается, но раздел `Монетки` может оставаться видимым по флагу `isCoinsEnabled` и показывать баланс. React не должен показывать disabled-заглушки для выключенных возможностей.

Архитектурное правило навигации: `Мой кошелёк` ведет себя как home/root action раздела, а не как noop для уже активного пункта меню. Реализация остается в React shell: `App.tsx` передает в `WalletPage` отдельный сигнал повторного перехода, а `WalletPage` сбрасывает внутренний state детализации бренда. Backend, Application use cases и wallet API для этого не меняются.

Компонентная модель кошелька: `WalletPage` отвечает за основной список наград по брендам, загрузку кошелька, актуализацию одноразового кода списания и переход в бренд; карточка бренда в секции `Мои награды` кликабельна целиком и открывает `BrandWalletPage`, отдельная кнопка `Открыть` внутри карточки не используется. `BrandWalletPage` отвечает за экран наград конкретного бренда; `RedemptionCodeCard` переиспользуется в основном кошельке и на экране бренда; `WalletBrandDetailsBlock` переиспользуется в экране бренда и в карточке клиента brand workspace. На экране бренда код списания расположен после плашки с названием бренда и до блока наград.

## Диагностика web-стенда

Для локальной диагностики подключены Serilog и Seq.

Архитектурно логирование настраивается в backend host-проектах:

- `StampService.API` пишет HTTP request logs, ошибки middleware, startup/shutdown и события `ILogger<T>`.
- `StampService.TelegramBot` пишет startup/shutdown, TelegramBotFlow pipeline logs и события `ILogger<T>`.
- `Application` и `Infrastructure` не должны зависеть от Serilog напрямую; они используют `ILogger<T>`.

Seq поднимается через `compose.yaml` вместе с инфраструктурой. Локальный адрес Seq: `http://localhost:5441`. Пароль первого запуска для локального стенда: `secure`.

В Seq полезно фильтровать события так:

- `Application = 'StampService.API'`
- `Application = 'StampService.TelegramBot'`
- `@Level = 'Error' or @Level = 'Fatal'`

Для web-сценариев важно: frontend не должен логировать JWT, OTP, технические ids, debug-поля и чувствительные данные. Ошибки UI должны оставаться пользовательскими, а техническая диагностика должна идти через backend logs и Seq.

Для бизнес-разборов используется отдельный `BusinessAuditLog`. Frontend не должен пытаться строить аудит из console logs или технических request logs: web показывает готовые audit DTO из backend, а сами audit-события создаются в Application use cases. Backend сохраняет audit records отдельно от основной business persistence boundary, чтобы журналирование не коммитило незавершенные бизнес-изменения. В audit/UI не выводятся JWT, OTP, redemption codes, raw metadata, internal ids и debug-поля.

TelegramBot при старте отправляет администраторам из `Admin:TelegramUserIds` диагностические сообщения со ссылками на web UI и Seq. URL задаются в `StartupNotifications` в конфигурации TelegramBot. Это удобство локального стенда, а не часть бизнес-логики web-приложения.

## Web: глобальная админка

В React web UI добавлен раздел `Админка`. Это отдельная глобальная административная поверхность, не часть workspace владельца/сотрудника бренда.

Назначение первого web-раздела админки:

- видеть список брендов;
- создавать бренд с владельцем по телефону;
- переназначать владельца бренда по телефону;
- смотреть журнал бизнес-операций для разбора начислений, списаний, выдач товаров, сотрудников и reward-настроек;
- выполнять стендовые demo-операции, аналогичные Telegram demo-flow.

Web-админка работает через `src/StampService.Web/src/admin/AdminPage.tsx` и typed API client `src/StampService.Web/src/admin/adminApi.ts`. Backend endpoints находятся в `src/StampService.API/Controllers/AdminController.cs`.

Текущая UI-модель админки:

- `AdminPage` - thin контейнер состояния, loading/error/status и auth reset после demo reset;
- верхний уровень - segmented tabs `Бренды` / `Журнал` / `Демо`;
- вкладка `Бренды` содержит локальный поиск и компактный список брендов в стиле `BrandSelector`;
- создание бренда и смена владельца открываются в bottom sheet (`CreateBrandSheet`, `ReassignOwnerSheet`), а не рендерятся постоянными формами на странице или внутри каждой карточки;
- карточка бренда показывает только пользовательские данные: название, владельца и телефон владельца; служебные ids и технические flags/chips не выводятся;
- кнопка создания бренда в toolbar - icon-only `+`, справа от поиска; отдельная кнопка обновления в текущем UI не показывается;
- вкладка `Журнал` показывает business audit cards с фильтрами и автоматической загрузкой при изменении фильтров;
- вкладка `Демо` содержит отдельные карточки для demo-брендов, demo-данных пользователю и опасного reset с подтверждением.

Визуально админка должна следовать тем же локальным паттернам, что `Мой кошелёк`, `BrandSelector` и `BrandWorkspace`: ширина около `760px`, выравнивание по левой границе workspace, белые карточки с тонкой границей, compact mobile-first списки, segmented controls и bottom sheets. Не нужно делать отдельную UI-библиотеку или параллельный набор компонентов ради админки.

Авторизация web-админки:

- пользователь входит в web по телефону через обычный OTP/JWT flow;
- backend проверяет текущий `User.Id` из JWT;
- Application `AdminAccessService` считает пользователя админом, если у него есть активная `Phone` identity из `Admin:PhoneNumbers`;
- текущий локальный номер администратора: `+79214408362`;
- Telegram-админка продолжает использовать `Admin:TelegramUserIds`; общий Application abstraction для обоих каналов - `AdminActor`.

Сценарии брендов:

- `GET /api/admin/access` - lightweight-проверка, можно ли показывать текущему web-пользователю глобальную админку;
- `GET /api/admin/brands` - загрузка списка брендов;
- `POST /api/admin/brands` - создание бренда с владельцем;
- `PUT /api/admin/brands/{brandId}/owner` - смена владельца.

Сценарий журнала операций:

- `GET /api/admin/audit-logs` - загрузка `BusinessAuditLog` как presentation-friendly списка для web-админки;
- typed client - `getBusinessAuditLogs` в `src/StampService.Web/src/admin/adminApi.ts`;
- фильтры журнала: период, бренд, клиент по телефону, исполнитель, тип операции, статус и лимит записей;
- поле `Исполнитель` - это поиск по человекочитаемому имени/отображаемому имени actor-а, а не ввод `UserId`;
- записи должны обновляться при изменении фильтров без обязательной кнопки `Показать`; для текстовых фильтров используется debounce, чтобы не стрелять запросом на каждый символ;
- телефон клиента отправляется в API только когда введен полный валидный номер; неполный телефон остается UX-валидацией frontend и не должен давать бессмысленный запрос;
- кнопка `Обновить` нужна для ручного refetch текущих фильтров, `Сбросить` возвращает журнал к базовому состоянию.

Правила:

- web не создает владельца автоматически;
- телефон владельца должен принадлежать существующему phone-first пользователю;
- создание бренда с владельцем должно быть атомарным на уровне persistence: frontend вызывает один API-сценарий, Application создает `Brand` и owner `BrandMembership`, а Infrastructure сохраняет их одним `SaveChanges`, чтобы не появлялся бренд без владельца;
- frontend только собирает ввод и показывает результат, бизнес-проверки остаются в Application;
- журнал операций показывает пользовательские summary/status/reason/comment и детали баланса/суммы, но не raw ids, trace ids, technical metadata и служебные коды.

## Web: demo-раздел админки

В админке добавлен раздел `Демо`, повторяющий основной Telegram demo-функционал:

- создать демо-бренды;
- создать демо-данные пользователю;
- очистить всю БД стенда.

HTTP API:

- `POST /api/admin/demo/brands`;
- `POST /api/admin/demo/user-data`;
- `POST /api/admin/demo/reset`.

Application use cases:

- `CreateDemoBrandsCommand`;
- `CreateUserDemoDataCommand`;
- `ResetDemoDatabaseCommand`.

Для web эти commands принимают `AdminActor.FromUser(userId)`, для Telegram - `AdminActor.FromTelegram(telegramUserId)`. Это сделано, чтобы не дублировать бизнес-логику и сохранить единый Application слой.

Важные UX/безопасность детали:

- `ResetDemoDatabaseCommand` удаляет в том числе `users` и `user_identities`, поэтому текущий JWT после reset больше невалиден;
- после успешной очистки web очищает auth-сессию и отправляет пользователя на новый вход;
- reset в UI требует явного текстового подтверждения `ОЧИСТИТЬ`;
- demo-сценарии остаются стендовыми/административными инструментами, не пользовательским функционалом клиента или бренда.
- создание demo brands остается одним Application use case: frontend вызывает `CreateDemoBrandsCommand`, а Application staged-создает бренды, owner memberships, метрики и товары и сохраняет их одним persistence boundary. Web не должен дробить это на несколько API-вызовов или локально управлять частями demo-brand creation.

Текущее правило развития интерфейса:

## Web: рабочая консоль бренда

Brand workspace в React web UI теперь проектируется как mobile-first рабочая консоль владельца/сотрудника, а не как длинная страница с несколькими независимыми panels. Актуальный рабочий flow начинается с выбора клиента: сотрудник сначала открывает карточку клиента по телефону, затем выполняет операции в контексте выбранной карточки.

Навигационная модель:

- первое состояние workspace - отдельный экран поиска клиента по телефону;
- поиск клиента не создает пользователя автоматически: это read-only открытие карточки существующего пользователя по активной `Phone` identity;
- если клиент не найден, `GET /api/brands/{brandId}/customer-card` должен вернуть обычный lookup-ответ `found=false`, а frontend открывает локальную draft-карточку нового клиента на основе введенного телефона; это не backend-сущность, не запись в БД и не исключение из read-only правила поиска;
- экран поиска может показывать таблицу `Недавние номера`: до 6 последних успешно открытых телефонов и дату открытия, сохраненные в `localStorage` отдельно для каждого `brandId`;
- `localStorage` в поиске клиента используется только как локальное UX-удобство для повторного открытия успешно открытых карточек, не заменяет backend-проверки доступа и не является бизнес-аудитом;
- второе состояние workspace - выбранный клиент: сверху карточка клиента, ниже операции над ним;
- обычная карточка клиента показывает имя, телефон и общий блок `Штампы` / `Монетки` / `История`;
- draft-карточка показывает `Новый клиент`, телефон и короткое пояснение, что истории пока нет, а клиент будет создан автоматически при первом начислении; в ней не показываются технические ids, `CustomerCode`, реальные балансы, история или wallet details;
- для draft-клиента доступны только действия первого начисления: выдать штампы и начислить монетки, с учетом `canIssue` и включенных feature toggles бренда; списания, выдача товара за монетки и история недоступны и не рендерятся;
- операции внутри выбранного клиента делятся на `Штампы` и `Монетки`; выдача товаров за монетки находится действием внутри `Монеток`;
- уровень действий показывается только при реальном выборе из нескольких действий;
- управление брендом не является верхней вкладкой рабочей зоны: оно открывается отдельным экраном по кнопке-шестерёнке в плашке бренда;
- внутри отдельного экрана настроек доступны `Штампы`, `Товары`, `Сотрудники`, `Бренд` по permissions;
- недоступные разделы не отображаются, а не показываются disabled.

Правила реализации:

- источник прав доступа - только `BrandWorkspaceResponse`: `canIssue`, `canRedeem`, `canViewBalances`, `canManageBrand`, `canManageMetrics`, `canManageStaff` и reward flags;
- frontend не переносит бизнес-правила из Application; локальные UX-кэши вроде недавних телефонов не должны хранить internal ids и не должны использоваться как источник истины;
- для открытия карточки клиента используется thin API `GET /api/brands/{brandId}/customer-card` и typed client `getBrandCustomerCard`;
- typed client получает lookup DTO (`found` + nullable `card`): `found=true` открывает обычную карточку, `found=false` открывает draft-карточку, а реальные ошибки доступа/валидации продолжают идти через общий API error flow;
- backend/Application нормализует телефон, проверяет доступ, ищет существующую активную `Phone` identity и возвращает карточку клиента; frontend не реализует поиск клиента сам и использует not found только как UX-сигнал для draft-карточки;
- существующие функции `getBrandWorkspace`, `getIssueMetricOptions`, `getRedeemMetricOptions`, `getCoinProductPurchaseOptions`, `issueMetricByPhone`, `redeemMetric`, `issueCoinsByPhone`, `redeemCoins`, `purchaseCoinProduct` остаются основной интеграционной поверхностью операций;
- management-сценарии штампов, товаров, сотрудников и настроек бренда переиспользуют текущие панели и API-вызовы;
- выдача штампов/монеток в выбранной карточке использует телефон выбранного клиента, чтобы сотрудник не вводил его повторно;
- начисление из draft-карточки использует те же typed API calls `issueMetricByPhone` и `issueCoinsByPhone`; после успешного ответа workspace повторно вызывает `getBrandCustomerCard` по телефону, заменяет draft на обычную карточку и добавляет телефон в `Недавние номера`;
- списание штампов, списание монеток и выдача товара за монетки всегда требуют одноразовый код списания клиента;
- UI-сортировка options допустима только как presentation-слой: при списании/выдаче сначала показываются доступные награды, затем недоступные с пользовательской причиной.

Компонентная модель:

- `BrandWorkspacePage` является контейнером: хранит выбранный `BrandWorkspaceResponse`, вызывает `getBrandWorkspace(brandId)` и переключает selector/workspace;
- `BrandSelector` отвечает только за список рабочих брендов, loading/error/empty states и открытие выбранного бренда;
- `BrandWorkspace` отвечает за рабочую область конкретного бренда, не содержит логики загрузки списка брендов и переключает внутренние экраны: поиск клиента, работа с клиентом, настройки бренда;
- `BrandCustomerSearchScreen` отвечает за экран поиска клиента по телефону, вызов `getBrandCustomerCard`, разделение not found/error states и локальную таблицу `Недавние номера`; при not found он сообщает наверх телефон для draft-карточки, не создавая клиента; recent-cache helper может переиспользоваться workspace после успешного сохранения draft-клиента через начисление;
- `SelectedCustomerWorkspace` внутри `BrandWorkspace` отвечает за экран работы с выбранным клиентом: обычную или draft-карточку клиента и операции над ней;
- `BrandSettingsPage` внутри `BrandWorkspace` отвечает за отдельный экран управления брендом и переиспользует management-панели;
- блок `Штампы` / `Монетки` / `История` в карточке клиента должен переиспользовать `WalletBrandDetailsBlock`, а не дублировать визуальную реализацию наград;
- `initialBrandId` сохраняет сценарий прямого открытия workspace, а `initialBrands` позволяют использовать уже загруженный список из `AppShell`;
- ошибка открытия workspace остается на экране выбора бренда, чтобы пользователь мог повторить действие или выбрать другой бренд.

Навигационное правило: `Рабочие бренды` / `Работа` ведет себя как home/root action раздела. Повторный клик по пункту меню должен открывать корневую форму раздела, даже если active section уже `brands`: при нескольких брендах это selector рабочих брендов, при одном бренде - заново открытый workspace этого бренда со сброшенным внутренним состоянием. Это реализуется сигналом из `App.tsx` в `BrandWorkspacePage`, без изменения API и Application. Локальные кнопки `Назад` в `BrandWorkspace` и во внутреннем `BrandSettingsPage` не используются; если потребуется другой способ возврата, его нужно проектировать как часть общей навигационной модели, а не возвращать разрозненные back-кнопки.

Визуальные договоренности:

- верх экрана: отдельная плашка/hero-зона бренда без локальной кнопки `Назад`;
- в плашке бренда показываются название, человекочитаемая роль и кнопка-шестерёнка для настроек, если управление доступно; flags/chips и технические поля не показываются;
- переключатели уровней оформляются как segmented controls;
- рабочая зона показывает один активный сценарий: поиск клиента или работа с найденным клиентом;
- карточками являются формы, options и результаты, без большой общей surface-panel вокруг всего workspace;
- плашка выбранного клиента не должна превращаться в toolbar: основные действия смены клиента/ручного обновления не закреплены в ней;
- selector рабочих брендов проектируется как компактный mobile-first список: avatar с первой буквой, название бренда, человекочитаемая роль и действие `Открыть`; raw `roleSystemName` не показывается;
- таблица `Недавние номера` находится под формой поиска, показывает телефон в той же маске, что phone input, дату открытия, действие повторного открытия карточки и действие очистки списка;
- стили находятся в `src/StampService.Web/src/styles.css`, scoped к brand workspace;
- основные компоненты находятся в `src/StampService.Web/src/brands/BrandWorkspacePage.tsx`, `BrandSelector.tsx`, `BrandWorkspace.tsx`, `BrandCustomerSearchScreen.tsx`; typed API client остается в `src/StampService.Web/src/brands/brandWorkspaceApi.ts`; общий wallet-блок наград находится в `src/StampService.Web/src/wallet/WalletBrandDetailsBlock.tsx`.

## Web: системные настройки SMS-кодов

В глобальной web-админке есть раздел `Настройки` для системных параметров, которые не относятся к конкретному бренду. Сейчас там управляется включение SMS-кодов входа.

UX-правило: бинарные системные настройки отображаются как тумблеры, а не как кнопки `Включить`/`Выключить`. Тумблер должен ясно показывать текущее состояние, иметь disabled/loading состояние на время запроса и не требовать отдельной кнопки сохранения для одиночной настройки.

Интеграционная модель:

- админка читает настройку через typed client `getPhoneAuthSmsSettings`;
- изменение идет через `updatePhoneAuthSmsSettings(isEnabled)`;
- API остается thin layer поверх Application command/query и проверки системного админа;
- frontend не хранит настройку как постоянный локальный source of truth и после ошибки должен вернуть UI к фактическому состоянию.

Эта настройка управляет только возможностью отправки OTP клиенту по SMS. Она не отключает базовую генерацию OTP и не отключает отправку кода администратору в Telegram для обычного dev/admin-сценария `Получить код`.
