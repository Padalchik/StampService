# StampService Web UI: рабочие договорённости

Актуально на 2026-05-25, Europe/Moscow.

Этот файл фиксирует договорённости по разработке полноценного web-интерфейса StampService, чтобы не переобсуждать базовые решения в новых задачах.

## Цель

Разработать полноценное React web-приложение для StampService.

В конечном состоянии web-приложение должно иметь функционал, аналогичный текущему Telegram-боту. При этом бизнес-логика не должна переноситься во frontend или API controllers: правила остаются в Domain/Application, а web/API остаются thin UI/API layer.

## Первая целевая аудитория

Первый web-сценарий покрывает:

- клиента;
- сотрудника бренда;
- владельца бренда.

Глобальная системная админка не является первым приоритетом, если она явно не запрошена отдельной задачей.

## Первый этап реализации

Первым этапом сделать только авторизацию по телефону.

Авторизация должна следовать существующим правилам приложения:

- вход по телефону через OTP;
- первичный вход/авторизация пользователя происходит только через OTP по телефону; сам аккаунт также может быть заранее создан бизнес-сценарием начисления по телефону;
- активный пользовательский аккаунт должен иметь активную `Phone` identity;
- OTP создаётся существующими Application-сценариями;
- доставка кода идёт через текущую реализацию `IPhoneAuthCodeSender`;
- пока SMS-провайдер не выбран, код приходит администратору в Telegram;
- подтверждение первичного телефонного входа должно опираться на актуальный активный OTP для нормализованного номера телефона, а не на технический id UI-сценария;
- после подтверждения кода frontend получает JWT;
- frontend не показывает пользователю OTP-коды, debug-данные, internal ids и служебные поля.

Telegram не является первичным способом создания аккаунта в web. Web может предлагать первичную привязку Telegram только из уже авторизованной телефонной сессии, если у пользователя еще нет активной Telegram identity.

Если при реализации возникнет неясность по правилам авторизации, нужно остановиться и уточнить правило у владельца проекта.

## Локальный запуск web-стенда

Для ручного тестирования web-приложения используется VS Code launch configuration:

- `App: полный запуск в браузере`

Эта конфигурация одной кнопкой запускает:

- `StampService.API`;
- `StampService.TelegramBot`;
- React/Vite frontend;
- браузер на странице входа web-приложения.

PostgreSQL не должен подниматься этой конфигурацией. Владелец проекта запускает PostgreSQL самостоятельно.

Для текущей телефонной авторизации важно, что OTP-код доставляется администратору в Telegram. Поэтому для web-входа должны быть выполнены условия:

- `StampService.API` запущен;
- `StampService.TelegramBot` запущен для обычной Telegram UX;
- Telegram bot token доступен конфигурации API, потому что отправку OTP из web-сценария выполняет API через `IPhoneAuthCodeSender`;
- `Admin:TelegramUserIds` настроен и содержит Telegram id администратора.

## Формат frontend

Web UI должен быть полноценным frontend-проектом на React.

Ориентир:

- React;
- TypeScript;
- нормальная структура приложения: роутинг, API-клиент, auth state, layout, страницы, общие компоненты;
- не развивать текущий `wwwroot/app.html` как основную долгосрочную web-поверхность.

Текущий static web UI в `src/StampService.API/wwwroot` можно использовать как справочный/временный прототип, но не как целевую архитектуру web-приложения.

## Визуальный стиль

На первом этапе приоритет - функциональность, ясность и устойчивость интерфейса, а не визуальная полировка.

Правила:

- интерфейс только на русском языке;
- тексты пользовательские, без технических формулировок;
- в web-навигации не использовать emoji: web должен быть спокойным рабочим интерфейсом, даже если в Telegram для тех же сценариев используются emoji;
- повторяемые названия разделов и навигации хранить в одном локальном frontend-источнике, а не дублировать строками по компонентам;
- не показывать пользователю ids, debug, internal codes и служебные поля;
- disabled/недоступные возможности бренда не должны протекать в UI;
- дизайн должен быть спокойным рабочим интерфейсом, пригодным для клиента, сотрудника и владельца бренда.

## Авторизация и сессия

Используется JWT, потому что backend уже поддерживает JWT.

Базовая модель сессии: пользователь входит по телефону, получает JWT для `User.Id`, у которого есть активная `Phone` identity. Telegram-login не должен создавать новый `User` без подтвержденного телефона; если такой endpoint сохраняется для совместимости, web не должен использовать его как первичный вход.

То же правило применяется к первичному входу через Telegram-бот: если Telegram identity еще не привязана, бот ведет пользователя через подтверждение телефона и затем привязывает Telegram к найденному или созданному телефонному аккаунту. Если аккаунт уже создан в web по телефону, вход в Telegram по этому же телефону должен привязать текущий Telegram id к тому же `User`. Такой phone-first onboarding концептуально равен телефонному входу, поэтому подтверждение OTP не должно зависеть от `AuthCodeId` из Telegram session. `AuthCodeId` допустим только для stateful-сценариев профиля, где identity еще не привязана.

Штатной перепривязки телефона или Telegram в web/Telegram UX нет. Если у пользователя уже есть активная identity нужного типа или внешний ключ принадлежит другому `User`, Application-сценарий должен отказать. Перенос identity между пользователями и миграция legacy Telegram-only аккаунтов допускаются только как отдельный явно согласованный flow.

Так как владелец проекта не хочет самостоятельно принимать слабые security-решения, реализация должна ориентироваться на best practices:

- изолировать работу с токеном в отдельном auth/API слое frontend;
- не размазывать JWT-логику по компонентам;
- централизованно обрабатывать `401 Unauthorized`;
- иметь явный logout;
- не логировать токен и чувствительные данные;
- хранение токена выбрать осознанно при реализации и зафиксировать причину в задаче/ответе.

Если потребуется изменить backend-авторизацию, например перейти на httpOnly cookie/refresh-token flow, это нужно обсуждать как отдельное архитектурное решение.

## Телефонные номера

Единое правило для телефонных номеров:

- пользовательский ввод должен приводиться к международному формату с `+`, например `+79991234567`;
- базовый синтаксис соответствует E.164: `+`, далее цифры, первая цифра не `0`, максимум 15 цифр;
- для номеров с кодом `+7` дополнительно требуется ровно 11 цифр вместе с кодом страны, например `+79991234567`;
- разрешены цифры и обычные разделители ввода: пробел, дефис, круглые скобки;
- буквы, несколько `+`, номер без `+`, слишком короткий или слишком длинный номер считаются ошибкой;
- внутри системы телефон хранится в нормализованном виде без разделителей, например `+79991234567`;
- backend остаётся авторитетным источником валидации;
- web и Telegram UI могут делать раннюю UX-проверку, но не должны обходить Domain/Application validation;
- в web-полях телефона используется маска для российских номеров: поле закрепляет префикс `+7`, пользователь вводит локальные цифры, UI форматирует значение как `+7 (999) 123-45-67`, а перед отправкой нормализует его в `+79991234567`;
- если пользователь начинает ввод с `7` или `8`, web-маска воспринимает это как попытку ввести код страны и не добавляет эти цифры в локальную часть номера.

Общее правило C#-валидации находится в доменной модели телефона, а Application-сценарии используют его через нормализацию перед созданием OTP и phone identity.

## Роли и навигация

Пока допустим левый многоуровневый sidebar/menu.

Ориентир навигации:

- клиентские разделы доступны пользователю как базовая часть приложения;
- если пользователь является сотрудником или владельцем бренда, показывать разделы рабочих брендов;
- действия в бренде показывать с учётом роли и feature toggles;
- недоступные действия лучше скрывать, как сейчас в Telegram UX, если нет отдельной причины показывать disabled-состояние.
- Основные web labels сейчас: `Мой кошелек`, `Рабочие бренды` и `Настройки аккаунта`. Повторяемые labels хранятся в `src/StampService.Web/src/app/navigationLabels.ts`. `Настройки аккаунта` располагаются внизу основной навигации.

## Telegram parity и API

Целевое правило: web должен постепенно повторить экраны и функционал Telegram-бота.

Важная архитектурная оговорка: Telegram-бот часто вызывает Application use cases напрямую, а не HTTP endpoints. Поэтому факт, что сценарий уже работает в Telegram, означает, что бизнес-сценарий в Application, скорее всего, уже есть. Но это не гарантирует, что для него уже есть HTTP endpoint, пригодный для web frontend.

Telegram parity не означает буквальное совпадение подписей кнопок. UI-copy может отличаться по каналу: Telegram может использовать emoji, web - нет. При этом внутри каждого UI названия должны быть централизованы локально.

Для клиентских операций начисления web и Telegram следуют одной продуктовой модели: метрики и монетки начисляются по номеру телефона клиента. Если клиент с такой активной `Phone` identity уже есть, начисление идет существующему `User`; если клиента еще нет, Application создает полноценного `User` с `Phone` identity и сразу проводит начисление через ledger. 4-значный `CustomerCode` не должен быть вводом в новых UI-сценариях начисления; старые HTTP/Application ветки начисления по `CustomerCode` удалены.

Важная граница: решение “найти или создать клиента по телефону” живет в Application use cases (`IssueMetricByPhoneCommand`, `IssueCoinsByPhoneCommand`), а не в React, API controller или Telegram endpoint. UI-слои только собирают телефон, количество и комментарий, затем вызывают общий сценарий. Это нужно, чтобы web и Telegram не расходились в поведении.

Списание остается отдельной моделью: метрики, ручное списание монеток и выдача товаров за монетки выполняются по одноразовому коду списания клиента, а не по телефону.

Просмотр клиентских балансов и истории операций сотрудником использует телефон клиента как внешний идентификатор, но не создает клиента автоматически. В отличие от начисления, это read-only сценарий: Application нормализует телефон, ищет существующую активную `Phone` identity и возвращает понятный отказ, если клиент не найден.

Управление сотрудниками бренда следует тому же phone-first направлению, но с другой бизнес-границей: добавление сотрудника не создает нового пользователя. Сотрудник/owner вводит телефон будущего сотрудника, Application нормализует номер, ищет существующий `User` по активной `Phone` identity и добавляет `BrandMembership` с ролью `STAFF`. Если пользователя с таким телефоном нет, сценарий должен отказать. Это сделано через `AddBrandStaffByPhoneCommand`; Telegram staff-flow уже использует этот подход и больше не вводит/не показывает `CustomerCode` для сотрудников. Когда аналогичный web UI будет добавляться, он должен быть thin UI поверх этого же Application-сценария, а не принимать `CustomerCode` или внутренний `UserId` как пользовательский ввод.

Системные админские сценарии назначения владельца бренда тоже переведены на телефон. Создание бренда с владельцем и смена владельца используют существующего пользователя с активной `Phone` identity: `CreateBrandWithOwnerCommand` принимает телефон владельца, `ReassignBrandOwnerCommand` - телефон нового владельца. Auto-create пользователя в этих сценариях запрещен, потому что это управление доступом. Telegram admin-flow уже следует этому правилу; если появится web/admin UI, он должен принимать телефон, а не `CustomerCode` или внутренний `UserId`.

Демо-инструменты также не должны возвращаться к `CustomerCode`. Создание пользовательских демо-данных выполняется по телефону существующего аккаунта через `CreateUserDemoDataCommand`: Application ищет активную `Phone` identity и наполняет выбранный бренд демо-балансами/историей через ledger-сервисы. Auto-create пользователя в demo-flow не выполняется. Если такой инструмент появится в web, UI должен принимать телефон и бренд, а не внутренние ids или старый код пользователя.

При разработке web-сценариев нужно:

1. Сначала смотреть, как соответствующий сценарий реализован в Telegram-боте.
2. Находить Application command/query, который использует Telegram.
3. Проверять, есть ли уже HTTP endpoint для этого сценария.
4. Если endpoint есть - использовать его.
5. Если endpoint отсутствует - сообщить владельцу проекта и согласовать добавление thin API endpoint поверх существующего Application use case.

Нельзя дублировать бизнес-логику из Telegram-бота во frontend.

## Web: рабочие бренды и клиентские операции

Раздел `Рабочие бренды` реализован как первая web-поверхность для сотрудника/owner. Цель раздела - дать web-аналог Telegram-механизма выдачи и списания метрик, монеток и товаров.

Архитектурная модель:

- frontend показывает список рабочих брендов текущего пользователя;
- открытие бренда загружает workspace: роль, permissions и feature toggles;
- UI скрывает действия, которые недоступны по роли или отключены настройками бренда;
- frontend не знает бизнес-правил балансов и не меняет их напрямую;
- API endpoints остаются тонким слоем поверх Application use cases;
- все операции с балансами проходят через существующие Application handlers и ledger-сервисы.

Текущие web-сценарии:

- выдача метрики по номеру телефона клиента;
- списание метрики по одноразовому коду списания клиента;
- начисление монеток по номеру телефона клиента;
- ручное списание монеток по одноразовому коду списания;
- выдача товара за монетки по одноразовому коду списания.

Управление сотрудниками бренда в web еще должно развиваться как отдельная рабочая поверхность owner/staff-management. Архитектурное правило уже определено Telegram/Application-реализацией: пользовательский ввод - телефон сотрудника; Application use case - `AddBrandStaffByPhoneCommand`; auto-create пользователя не выполняется; отображение сотрудников должно использовать телефон, имя и роль, но не `CustomerCode`.

Важное отличие web от Telegram: Telegram flow хранит промежуточное состояние в bot session, а web делает обычные HTTP-запросы из React state. Поэтому для web добавлены endpoints, которые принимают web-friendly входные данные. Например, выдача метрики принимает номер телефона клиента и передает его в `IssueMetricByPhoneCommand`; Application сам нормализует телефон, находит или создает клиента и проводит начисление. Web не должен принимать или показывать внутренний `UserId` клиента.

Для web-форм начисления телефонная маска используется как UX-помощник, а не как источник бизнес-правил. Frontend отправляет нормализованный телефонный номер в thin endpoint, backend/Application повторно валидирует номер и выполняет общий issue-by-phone flow.

Ключевые API endpoints:

- `GET /api/brands/mine`;
- `GET /api/brands/{brandId}/workspace`;
- `GET /api/brands/{brandId}/metrics/issue-options`;
- `GET /api/brands/{brandId}/metrics/redeem-options?redemptionCode=...`;
- `POST /api/metrics/{metricDefinitionId}/issue-by-phone` - web-friendly thin endpoint поверх `IssueMetricByPhoneCommand`;
- `POST /api/metrics/{metricDefinitionId}/redeem`;
- `POST /api/brands/{brandId}/coins/issue-by-phone` - web-friendly thin endpoint поверх `IssueCoinsByPhoneCommand`;
- `POST /api/brands/{brandId}/coins/redeem`;
- `GET /api/brands/{brandId}/coin-products/purchase-options?redemptionCode=...`;
- `POST /api/brands/{brandId}/coin-products/{productId}/purchase`.

Ключевые frontend места:

- `src/StampService.Web/src/brands/BrandWorkspacePage.tsx` - экран рабочих брендов, workspace и формы операций;
- `src/StampService.Web/src/brands/brandWorkspaceApi.ts` - typed API calls;
- `src/StampService.Web/src/validation/phoneNumber.ts` - frontend-маска и нормализация телефонного ввода;
- `src/StampService.Web/src/app/App.tsx` - подключение раздела в основной layout;
- `src/StampService.Web/src/styles.css` - стили раздела.

## MVP-порядок

Согласованный порядок развития web-приложения:

1. Скелет web app: React-проект, роутинг, auth guard, API client, layout, обработка ошибок.
2. Телефонный вход.
3. Профиль, первичная привязка телефона при необходимости и привязка Telegram.
4. Кошелёк клиента с кодом списания, наградами и историей.
5. Brand workspace для сотрудника/owner. Базовая клиентская работа уже добавлена: выдача/списание метрик, монеток и товаров через thin API + Application use cases.
6. Управление метриками, товарами, сотрудниками и настройками бренда.

## Web: личный кабинет

Экран `Личный кабинет` в web должен соответствовать данным профиля Telegram-бота и использовать общие Application-сценарии.

Архитектурное правило личного кабинета: телефон является обязательной первичной identity активного аккаунта, Telegram - вторичная привязка. Сценарии профиля не должны создавать Telegram-only аккаунты и не должны перепривязывать уже активные phone/Telegram identity.

Текущая реализация:

- frontend загружает профиль через `GET /api/users/me`;
- первичная привязка телефона использует существующие тонкие endpoints `POST /api/users/me/phone/code` и `POST /api/users/me/phone/verify`, если у профиля еще нет активного телефона;
- первичная привязка Telegram в web идёт через deep link, а не через ввод Telegram id или username;
- frontend запрашивает ссылку через `POST /api/users/me/telegram/link`;
- API возвращает ссылку вида `https://t.me/<bot_username>?start=<token>`;
- пользователь открывает Telegram по ссылке, TelegramBot получает `/start <token>` и привязывает реальный Telegram user id к текущему web-пользователю через Application-сценарий;
- для генерации ссылки API должен иметь `Telegram:BotUsername`, а для обработки deep link должен быть запущен `StampService.TelegramBot`;
- username Telegram может показываться как display value, но не используется как устойчивый ключ привязки.

Правило привязки identity:

- если пользователь входит по телефону, Application находит или создает единственный телефонный `User`;
- если пользователь привязывает Telegram из web, Telegram identity добавляется к текущему телефонному `User` только при отсутствии активной Telegram identity у этого `User`;
- если телефон или Telegram уже принадлежит другому `User`, обычный web-сценарий должен отказать с понятной причиной;
- если у текущего `User` уже есть активная identity этого типа, web не показывает действие перепривязки, а backend дополнительно отказывает при прямом вызове endpoint;
- автоматическое объединение аккаунтов и перенос `UserIdentity` между пользователями не являются базовым пользовательским сценарием новой модели, потому что новые Telegram-only аккаунты больше не должны создаваться;
- если legacy/migration flow все же нужен, перенос наград и балансов должен идти через Application/ledger-действия, а не прямым изменением балансов.

## Web: мой кошелёк

Экран `Мой кошелёк` в web должен соответствовать данным Telegram-бота.

Текущая реализация:

- frontend вызывает `POST /api/wallet/open`;
- API endpoint должен быть тонким и вызывать Application-сценарий;
- общий Application-сценарий для web и Telegram - `OpenUserWallet`;
- `OpenUserWallet` собирает данные через существующие use cases:
  - создаёт или возвращает актуальный код списания;
  - загружает overview балансов и доступных наград;
  - возвращает код пользователя, код списания, срок действия кода и бренды;
  - best-effort отмечает открытие кошелька для digest/уведомлений.

Правило: вспомогательная отметка открытия кошелька не должна ломать отображение кошелька. Если marker/digest state не сохранился, основной экран кошелька всё равно должен загрузиться, а проблема должна логироваться.

Для обновления кода списания используется тот же endpoint с `forceRefreshCode=true`.

React dev mode может вызывать effects повторно, поэтому initial-загрузка кошелька во frontend должна быть защищена от двойного запроса.

Детализация бренда в web-кошельке:

- из списка брендов пользователь нажимает `Подробнее`;
- детализация открывается как отдельный экран внутри раздела `Мой кошелёк`, а не как блок ниже списка;
- на экране детализации есть кнопка `Назад`, возвращающая к списку брендов;
- данные детализации должны соответствовать Telegram-сценарию:
  - доступные товары за монетки;
  - доступные метрики;
  - статус `доступно` / `не хватает`;
  - последние операции по монетам и метрикам;
- Web и Telegram используют единый Application-сценарий детализации бренда: `GetUserWalletBrandDetailsQuery`;
- Web получает детализацию через тонкий endpoint `GET /api/wallet/brands/{brandId}/details`;
- отдельные web/API контракты для rewards и history не являются текущей поверхностью Web UI: данные наград и последние операции собираются в одном details-сценарии;
- UI-слои остаются разными: Application возвращает presentation-friendly DTO без Telegram HTML, React JSX и CSS, а Web и Telegram только рендерят этот DTO в свой формат.

## Рабочие ограничения

- Перед изменениями читать существующий код и следовать локальным паттернам.
- Не запускать backend, Telegram bot, build, tests, migrations или dev servers без явного разрешения владельца проекта.
- Ручные изменения файлов делать через `apply_patch`.
- Не откатывать чужие изменения в рабочем дереве без явного разрешения.
- Для EF migrations не писать файлы вручную: использовать `dotnet ef migrations add`.

## Диагностика web-стенда

Для локальной диагностики подключены Serilog и Seq.

Архитектурно логирование настраивается в backend host-проектах:

- `StampService.API` пишет HTTP request logs, ошибки middleware, startup/shutdown и события `ILogger<T>`.
- `StampService.TelegramBot` пишет startup/shutdown, TelegramBotFlow pipeline logs и события `ILogger<T>`.
- `Application` и `Infrastructure` не должны зависеть от Serilog напрямую; они используют `ILogger<T>`.

Seq поднимается через `compose.yaml` вместе с инфраструктурой. Локальный адрес Seq: `http://localhost:5441`. Пароль первого запуска для локального стенда: `secure`.

В Seq полезно фильтровать события так:

- `Application = 'StampService.API'`
- `Application = 'StampService.TelegramBot'`
- `@Level = 'Error' or @Level = 'Fatal'`

Для web-сценариев важно: frontend не должен логировать JWT, OTP, технические ids, debug-поля и чувствительные данные. Ошибки UI должны оставаться пользовательскими, а техническая диагностика должна идти через backend logs и Seq.

TelegramBot при старте отправляет администраторам из `Admin:TelegramUserIds` диагностические сообщения со ссылками на web UI и Seq. URL задаются в `StartupNotifications` в конфигурации TelegramBot. Это удобство локального стенда, а не часть бизнес-логики web-приложения.
