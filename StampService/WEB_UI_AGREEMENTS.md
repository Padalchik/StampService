# StampService Web UI: рабочие договорённости

Актуально на 2026-05-18, Europe/Moscow.

Этот файл фиксирует договорённости по разработке полноценного web-интерфейса StampService, чтобы не переобсуждать базовые решения в новых задачах.

## Цель

Разработать полноценное React web-приложение для StampService.

В конечном состоянии web-приложение должно иметь функционал, аналогичный текущему Telegram-боту. При этом бизнес-логика не должна переноситься во frontend или API controllers: правила остаются в Domain/Application, а web/API остаются thin UI/API layer.

## Первая целевая аудитория

Первый web-сценарий покрывает:

- клиента;
- сотрудника бренда;
- владельца бренда.

Глобальная системная админка не является первым приоритетом, если она явно не запрошена отдельной задачей.

## Первый этап реализации

Первым этапом сделать только авторизацию по телефону.

Авторизация должна следовать существующим правилам приложения:

- вход по телефону через OTP;
- OTP создаётся существующими Application-сценариями;
- доставка кода идёт через текущую реализацию `IPhoneAuthCodeSender`;
- пока SMS-провайдер не выбран, код приходит администратору в Telegram;
- после подтверждения кода frontend получает JWT;
- frontend не показывает пользователю OTP-коды, debug-данные, internal ids и служебные поля.

Если при реализации возникнет неясность по правилам авторизации, нужно остановиться и уточнить правило у владельца проекта.

## Локальный запуск web-стенда

Для ручного тестирования web-приложения используется VS Code launch configuration:

- `App: полный запуск в браузере`

Эта конфигурация одной кнопкой запускает:

- `StampService.API`;
- `StampService.TelegramBot`;
- React/Vite frontend;
- браузер на странице входа web-приложения.

PostgreSQL не должен подниматься этой конфигурацией. Владелец проекта запускает PostgreSQL самостоятельно.

Для текущей телефонной авторизации важно, что OTP-код доставляется администратору в Telegram. Поэтому для web-входа должны быть выполнены условия:

- `StampService.API` запущен;
- `StampService.TelegramBot` запущен для обычной Telegram UX;
- Telegram bot token доступен конфигурации API, потому что отправку OTP из web-сценария выполняет API через `IPhoneAuthCodeSender`;
- `Admin:TelegramUserIds` настроен и содержит Telegram id администратора.

## Формат frontend

Web UI должен быть полноценным frontend-проектом на React.

Ориентир:

- React;
- TypeScript;
- нормальная структура приложения: роутинг, API-клиент, auth state, layout, страницы, общие компоненты;
- не развивать текущий `wwwroot/app.html` как основную долгосрочную web-поверхность.

Текущий static web UI в `src/StampService.API/wwwroot` можно использовать как справочный/временный прототип, но не как целевую архитектуру web-приложения.

## Визуальный стиль

На первом этапе приоритет - функциональность, ясность и устойчивость интерфейса, а не визуальная полировка.

Правила:

- интерфейс только на русском языке;
- тексты пользовательские, без технических формулировок;
- не показывать пользователю ids, debug, internal codes и служебные поля;
- disabled/недоступные возможности бренда не должны протекать в UI;
- дизайн должен быть спокойным рабочим интерфейсом, пригодным для клиента, сотрудника и владельца бренда.

## Авторизация и сессия

Используется JWT, потому что backend уже поддерживает JWT.

Так как владелец проекта не хочет самостоятельно принимать слабые security-решения, реализация должна ориентироваться на best practices:

- изолировать работу с токеном в отдельном auth/API слое frontend;
- не размазывать JWT-логику по компонентам;
- централизованно обрабатывать `401 Unauthorized`;
- иметь явный logout;
- не логировать токен и чувствительные данные;
- хранение токена выбрать осознанно при реализации и зафиксировать причину в задаче/ответе.

Если потребуется изменить backend-авторизацию, например перейти на httpOnly cookie/refresh-token flow, это нужно обсуждать как отдельное архитектурное решение.

## Телефонные номера

Единое правило для телефонных номеров:

- пользователь вводит номер в международном формате с `+`, например `+7 999 123-45-67`;
- базовый синтаксис соответствует E.164: `+`, далее цифры, первая цифра не `0`, максимум 15 цифр;
- для номеров с кодом `+7` дополнительно требуется ровно 11 цифр вместе с кодом страны, например `+79991234567`;
- разрешены цифры и обычные разделители ввода: пробел, дефис, круглые скобки;
- буквы, несколько `+`, номер без `+`, слишком короткий или слишком длинный номер считаются ошибкой;
- внутри системы телефон хранится в нормализованном виде без разделителей, например `+79991234567`;
- backend остаётся авторитетным источником валидации;
- web и Telegram UI могут делать раннюю UX-проверку, но не должны обходить Domain/Application validation.

Общее правило C#-валидации находится в доменной модели телефона, а Application-сценарии используют его через нормализацию перед созданием OTP и phone identity.

## Роли и навигация

Пока допустим левый многоуровневый sidebar/menu.

Ориентир навигации:

- клиентские разделы доступны пользователю как базовая часть приложения;
- если пользователь является сотрудником или владельцем бренда, показывать разделы рабочих брендов;
- действия в бренде показывать с учётом роли и feature toggles;
- недоступные действия лучше скрывать, как сейчас в Telegram UX, если нет отдельной причины показывать disabled-состояние.

## Telegram parity и API

Целевое правило: web должен постепенно повторить экраны и функционал Telegram-бота.

Важная архитектурная оговорка: Telegram-бот часто вызывает Application use cases напрямую, а не HTTP endpoints. Поэтому факт, что сценарий уже работает в Telegram, означает, что бизнес-сценарий в Application, скорее всего, уже есть. Но это не гарантирует, что для него уже есть HTTP endpoint, пригодный для web frontend.

При разработке web-сценариев нужно:

1. Сначала смотреть, как соответствующий сценарий реализован в Telegram-боте.
2. Находить Application command/query, который использует Telegram.
3. Проверять, есть ли уже HTTP endpoint для этого сценария.
4. Если endpoint есть - использовать его.
5. Если endpoint отсутствует - сообщить владельцу проекта и согласовать добавление thin API endpoint поверх существующего Application use case.

Нельзя дублировать бизнес-логику из Telegram-бота во frontend.

## MVP-порядок

Согласованный порядок развития web-приложения:

1. Скелет web app: React-проект, роутинг, auth guard, API client, layout, обработка ошибок.
2. Телефонный вход.
3. Профиль и смена телефона.
4. Кошелёк клиента с кодом списания, наградами и историей.
5. Brand workspace для сотрудника/owner.
6. Управление метриками, товарами, сотрудниками и настройками бренда.

## Web: мой кошелёк

Экран `Мой кошелёк` в web должен соответствовать данным Telegram-бота.

Текущая реализация:

- frontend вызывает `POST /api/wallet/open`;
- API endpoint должен быть тонким и вызывать Application-сценарий;
- общий Application-сценарий для web и Telegram - `OpenUserWallet`;
- `OpenUserWallet` собирает данные через существующие use cases:
  - создаёт или возвращает актуальный код списания;
  - загружает overview балансов и доступных наград;
  - возвращает код пользователя, код списания, срок действия кода и бренды;
  - best-effort отмечает открытие кошелька для digest/уведомлений.

Правило: вспомогательная отметка открытия кошелька не должна ломать отображение кошелька. Если marker/digest state не сохранился, основной экран кошелька всё равно должен загрузиться, а проблема должна логироваться.

Для обновления кода списания используется тот же endpoint с `forceRefreshCode=true`.

React dev mode может вызывать effects повторно, поэтому initial-загрузка кошелька во frontend должна быть защищена от двойного запроса.

Детализация бренда в web-кошельке:

- из списка брендов пользователь нажимает `Подробнее`;
- детализация открывается как отдельный экран внутри раздела `Мой кошелёк`, а не как блок ниже списка;
- на экране детализации есть кнопка `Назад`, возвращающая к списку брендов;
- данные детализации должны соответствовать Telegram-сценарию:
  - доступные товары за монетки;
  - доступные метрики;
  - статус `доступно` / `не хватает`;
  - последние операции по монетам и метрикам;
- API остаётся тонким и использует существующие Application queries:
  - `GetUserBrandRewardsQuery`;
  - `GetUserBrandWalletHistoryQuery`.

## Рабочие ограничения

- Перед изменениями читать существующий код и следовать локальным паттернам.
- Не запускать backend, Telegram bot, build, tests, migrations или dev servers без явного разрешения владельца проекта.
- Ручные изменения файлов делать через `apply_patch`.
- Не откатывать чужие изменения в рабочем дереве без явного разрешения.
- Для EF migrations не писать файлы вручную: использовать `dotnet ef migrations add`.
