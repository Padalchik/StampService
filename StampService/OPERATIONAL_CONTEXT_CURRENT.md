# Operational Context For Next Chat

Дата составления: 2026-05-13, Europe/Moscow.

## 1. Что за проект

StampService - .NET-сервис лояльности с основным UX через Telegram-бота. MVP нужен для бизнеса: создавать бренды, работать с клиентами, начислять/списывать клиентские бонусы, смотреть балансы и историю без тяжёлой админки.

Архитектура слоистая:

- `StampService.Domain` - доменные сущности и правила.
- `StampService.Application` - use cases, commands/queries, ошибки, доступы.
- `StampService.Infrastructure` - EF Core, репозитории, конфигурации, миграции.
- `StampService.API` - HTTP API.
- `StampService.TelegramBot` - Telegram UX и handlers.
- `external/telegram-bot-flow` - локальная библиотека Telegram flow/navigation.
- `tests/*` - Application/Domain tests.

Ключевые бизнес-сущности:

- `User` - общий пользователь системы.
- `UserIdentity` - внешние идентичности пользователя, сейчас важна Telegram identity. Архитектура уже разделяет пользователя и способ входа.
- `Brand` - бренд/бизнес.
- `BrandMembership` - роль пользователя в бренде.
- `LoyaltyMetricDefinition` - метрика лояльности бренда, например визит/кофе/стрижка.
- `MetricBalance` / `MetricTransaction` - баланс и ledger по метрикам.
- `CoinWallet` / `CoinTransaction` - универсальные монетки бренда.
- `CoinProduct` - товары бренда, которые можно купить за монетки.
- `RedemptionCode` - одноразовый код списания, который клиент показывает сотруднику.

## 3. Текущий UX и правила

### Главное меню

В главном меню:

- `Мой кошелек`
- `Админка`, если пользователь админ
- если один рабочий бренд - кнопка бренда
- если несколько брендов - `Рабочие бренды`

На самом главном меню не должно быть кнопки `Главное меню`.

### Мой кошелек

`Мой кошелек` показывает:

- `Код пользователя` вместо `CustomerCode`
- одноразовый код списания
- срок действия кода в локальном времени пользователя/окружения, без текста `UTC+03:00`
- балансы по брендам
- монетки с большой буквы там, где пользователь это просил
- кнопку `🔄 Обновить данные`

Важное правило: `Обновить данные` в кошельке должен обновлять код списания по правилам ротации. Если был активный неиспользованный код, он истекает, и его комбинация снова может выпасть кому-то другому.

### Бренд

Внутри бренда есть разделы:

- `Работа с клиентами`
- `Работа с метриками`
- `Товары за монетки`
- `Сотрудники`

При выборе бренда endpoint собирает доступные разделы. Если доступен ровно один раздел, бот открывает его сразу; если больше одного - показывает экран бренда.

Кнопки `К бренду` убраны из Telegram UX. Не добавлять их обратно без явного запроса.

Кнопки `Назад` пользователь просил убрать/минимизировать; в коде `.BackButton()` ещё встречается, но ранее менялось поведение/отображение через TelegramBotFlow. Перед массовой правкой проверить текущую реализацию библиотеки.

### Работа с клиентами

Кнопки:

- `🟢 Выдать метрику`
- `🟡 Списать метрику`
- `🟢 Начислить монетки`
- `🟡 Списать монетки`
- `Балансы клиента`

Важно: кнопка `🟡 Списать монетки` теперь ведёт не в ручное списание монеток, а в сценарий покупки товара за монетки.

Ручная бизнес-функция произвольного списания монеток удалена. Её заменяет покупка товара.

## 4. Метрики

Метрики бренда реализованы:

- список метрик
- детали метрики
- создание
- редактирование
- начисление клиенту
- списание
- история
- балансы клиента

`служебное название`/`code` метрики удалено из системы. В `LoyaltyMetricDefinition` остались пользовательское название и `RedemptionAmount`.

При создании метрики:

- `Введите название для отображения пользователям`
- нет ввода служебного кода

При редактировании метрики:

- любой ввод означает изменение
- для сохранения старого значения используется кнопка `Оставить эти данные`

После создания/редактирования метрики открывается экран `Метрики`.

При входе в `Работа с метриками` сразу открывается экран `Все метрики`; отдельный промежуточный экран убран/обойдён. Кнопка создания называется `➕ Создать новую метрику`.

## 5. Монетки

Монетки (`Coin`) - отдельный поддомен, не метрика.

Реализовано:

- `CoinWallet`
- `CoinTransaction`
- начисление монеток
- история монеток
- баланс монеток в кошельке и в балансах клиента
- `ActorUserId` в coin/stamp/metric transaction для технического аудита

Важно:

- `ActorUserId` не показывать пользователю в истории транзакций. Это техническая информация.
- Ручное произвольное списание монеток удалено:
  - удалены `RedeemCoinsCommand` / `RedeemCoinsHandler`
  - удалены Telegram actions/screens для ручного списания монеток
  - `CoinEndpoint` отвечает только за начисление и историю
- Списание монеток происходит через покупку товара за монетки.

Комментарий в coin transaction при покупке товара должен быть просто названием товара, без префикса `Purchase product:`.

## 6. Товары за монетки

Добавлена фича `CoinProduct`: товары, которые бренд может продавать за монетки.

Доменные/инфраструктурные файлы:

- `src/StampService.Domain/Coins/CoinProduct.cs`
- `src/StampService.Infrastructure/Configurations/CoinProductConfiguration.cs`
- `src/StampService.Infrastructure/Repositories/CoinProductRepository.cs`
- `ICoinProductRepository`
- DTOs в `src/StampService.Contracts/DTOs/CoinProducts/`
- handlers/queries в `src/StampService.Application/CoinProducts/`

Миграция создана через EF:

- `src/StampService.Infrastructure/Migrations/20260513105958_AddCoinProducts.cs`

Таблица:

- `coin_products`
- `brand_id`
- `name`
- `price`
- `is_active`
- audit-поля
- check constraint `price > 0`
- индекс `ix_coin_products_brand_id`

Удаление товара сейчас реализовано как деактивация (`IsActive = false`), не физическое удаление.

Права:

- управление товарами использует тот же доступ, что управление метриками: `PermissionCode.MetricManage`
- покупка товара использует `PermissionCode.StampRedeem`

API:

- `src/StampService.API/Controllers/CoinProductsController.cs`
- CRUD товаров
- purchase-options
- purchase

Telegram UX товаров:

- В бренде есть `Товары за монетки`.
- Список товаров показывает активные/выключенные товары.
- Создание/редактирование/удаление по аналогии с метриками.
- В `Работа с клиентами` кнопка `🟡 Списать монетки` открывает покупку товара за монетки.

Покупка товара:

1. сотрудник вводит код списания клиента;
2. бот показывает товары;
3. кнопки товаров показывают `сколько есть/сколько нужно`, например `Кофе · 7/10`;
4. если монеток не хватает, кнопка с `⛔️` остаётся на том же экране;
5. если хватает, выбор товара открывает экран подтверждения, как при списании метрики;
6. фактическое списание происходит только по `✅ Подтвердить`;
7. `❌ Отмена` отменяет сценарий.

Ключевые Telegram файлы:

- `Features/CoinProducts/Endpoints/CoinProductEndpoint.cs`
- `PurchaseCoinProductCodeScreen`
- `PurchaseCoinProductSelectScreen`
- `PurchaseCoinProductConfirmScreen`
- `SelectPurchaseCoinProductAction/Payload`
- `PurchaseCoinProductAction/Payload`
- `CancelPurchaseCoinProductAction`

## 7. Код пользователя и CustomerCode

Техническое имя `CustomerCode` в коде/модели оставлено.

В пользовательском UX использовать `Код пользователя`, не `CustomerCode` и не `клиентский код`.

Это уже применялось:

- `Мой кошелек`
- создание бренда/назначение владельца
- смена владельца
- работа с клиентом
- балансы/монетки/сотрудники

Если где-то в отображении ещё встретится `CustomerCode`, заменить на `Код пользователя`.

## 8. История транзакций

История должна оставаться пользовательской:

- не показывать `ActorUserId`
- не писать технические поля
- для кнопок истории использовать эмодзи `📈`

Для истории монеток:

- начисление: зелёный маркер
- списание: жёлтый маркер
- комментарий показывать, если он не авто-комментарий
- при покупке товара комментарий равен названию товара, поэтому его можно показывать как название купленного товара

## 9. TelegramBotFlow

Локальная библиотека: `external/telegram-bot-flow`.

Ранее правилось поведение navigation/back:

- action-view отличён от обычного screen stack;
- Back после transient/action view должен закрывать action-view/re-render текущий screen, а не делать лишний pop;
- pending input должен обновляться при render/show view.

Если UX внезапно ведёт назад в главное меню или старый экран, сначала проверить `external/telegram-bot-flow`.
