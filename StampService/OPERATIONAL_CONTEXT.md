# Operational Context For Next Chat

Дата составления: 2026-05-12, Europe/Moscow.

## 1. Что за проект

StampService - система лояльности, которая сейчас активно собирается вокруг Telegram-бота. Цель MVP: дать бизнесу простой инструмент для выдачи, списания и учета клиентских бонусов без сложной админки и без тяжелого фронтенда.

Основная бизнес-идея:

- У бизнеса есть бренды.
- У бренда есть владелец и сотрудники.
- Клиенты идентифицируются через `CustomerCode`, короткий публичный код пользователя.
- В бренде можно заводить метрики лояльности: например "кофе", "визит", "стрижка", "котик".
- Метрики можно начислять клиенту и списывать по правилам бренда.
- Помимо метрик добавлены "монетки" (`Coin`) - универсальный бонусный баланс, который можно начислять и списывать независимо от конкретной метрики.
- Telegram-бот является основным UX для MVP: через него пользователь видит свой кошелек, а владелец/сотрудник работает с клиентами и метриками.

Проект на .NET, слоистая архитектура:

- `StampService.Domain` - доменные сущности и правила.
- `StampService.Application` - use cases, команды, queries, ошибки, доступы.
- `StampService.Infrastructure` - EF Core, репозитории, конфигурации, миграции.
- `StampService.API` - HTTP API.
- `StampService.TelegramBot` - Telegram bot UX и handlers.
- `external/telegram-bot-flow` - локальная библиотека Telegram flow/navigation, уже правилась в рамках проекта.
- `tests/*` - тесты Application/Domain.

## 3. Архитектурные/UX принципы, которые уже согласованы

- Админка не должна управлять организационной операционкой бренда, если это не админский bootstrap. Например "Сотрудники" должны быть доступны владельцу бренда, а не глобальному админу как обычное действие.
- Владелец бренда управляет сотрудниками.
- При переназначении владельца старый владелец теряет membership в бренде.
- Владельца бренда назначают при создании бренда по `CustomerCode`.
- Админов Telegram-бота можно хранить в appsettings через `Admin`/`Admins`; текущий Telegram ID пользователя: `278225388`.
- Для списания метрики UX: сначала выбрать пользователя/код списания, затем показать метрики, которые можно списать. Недоступные метрики показываются как недоступные/ничего не делают при клике. В кнопках нужен формат с доступностью, например `⛔️ <метрика> <есть>/<нужно>`.
- В `BuildBalancesView` не использовать галочки/паузу эмодзи для активных/неактивных метрик. Если активные метрики есть - показывать их. Если нет активных - не показывать.
- В истории транзакций:
  - начисление: `🟢`
  - списание: `🟡`
  - эмодзи стоит перед текстом.
  - строка должна быть самодостаточной: дата, знак, количество, что именно изменилось, комментарий.
- В кнопках начисления/списания использовать такие же эмодзи:
  - `🟢 Выдать метрику`
  - `🟡 Списать метрику`
  - `🟢 Начислить монетки`
  - `🟡 Списать монетки`
- "Назад" должен реально возвращать назад, а не часто в главное меню. Библиотека была поправлена под это.
- Все кнопки подтверждения/отмены должны быть единообразными:
  - `✅ Подтвердить`
  - `❌ Отмена`

## 4. Что уже реализовано крупными блоками

### 4.1 Метрики

Реализованы:

- Просмотр метрик бренда.
- Проваливание в детали метрики.
- Создание метрики.
- Редактирование метрики.
- Начисление метрики клиенту.
- Списание метрики.
- История операций по метрике.
- Балансы клиента по метрикам.

Для списания метрики UX доработан:

- Сначала вводится/выбирается пользователь через код списания.
- Затем показываются метрики для списания.
- В тексте сообщения не дублируется список метрик, информация о достаточности баланса находится в кнопках.
- Формат кнопки должен быть `⛔️ <метрика> <есть>/<нужно>` для недоступных.

### 4.2 Бренды и владельцы

Реализованы:

- Создание бренда.
- Назначение владельца при создании по `CustomerCode`.
- Переназначение владельца.
- При переназначении старый владелец теряет membership.
- Админский доступ через конфигурацию.

### 4.3 Сотрудники

Реализованы:

- Назначение сотрудников бренда владельцем.
- Удаление сотрудников.
- Роли/права пока не раскрыты в UX как permission-настройки. Это сознательное решение: сейчас есть роли, но архитектурно нужно помнить, что в будущем права могут стать более детальными.
- Кнопка `Сотрудники` должна быть в контексте бренда для владельца, не в глобальной админке.

### 4.4 Монетки / Coin

Согласованное решение:

- Монетки не являются метрикой.
- Это отдельная сущность/поддомен `Coin`, потому что у метрики есть `RedemptionAmount` и специфическая логика списания, а монетки - универсальный бонусный баланс.
- В неймингах использовать `Coin`, не `Bonus`, потому что "Bonus" было слишком размыто.

Реализованы:

- Domain/Application/Infrastructure для Coin.
- `CoinWallet`.
- `CoinTransaction`.
- Начисление монеток.
- Списание монеток.
- История монеток.
- Баланс монеток в клиентских балансах.
- Монетки отображаются в `Мой кошелек` и в балансах клиента.

### 4.5 Кошелек

Последний согласованный UX:

- В главном меню вместо `Мой код`, `Код для списания`, `Мои балансы` теперь одна кнопка `Мой кошелек`.
- `Мой кошелек` показывает:
  - `CustomerCode`
  - свежий код для списания
  - срок действия кода
  - балансы по брендам
- Балансы отображаются иерархично:

```text
• <бренд>
  - <метрика> <сколько есть>/<сколько нужно>
  - монетки <значение>

• <бренд>
  - <метрика> <сколько есть>/<сколько нужно>
  - монетки <значение>
```

- Между блоками брендов должна быть пустая строка.
- В `Мой кошелек` больше не показывать отдельные кнопки истории по каждой метрике.
- Вместо этого показывать историю по брендам: `История: <бренд>`.
- История бренда должна объединять операции по метрикам и монеткам.

## 5. Текущий UX главного меню и бренда

### Главное меню

- `Мой кошелек`
- `Админка`, если пользователь админ
- если один бренд, кнопка вида `Бренд: <brandName>`

### Бренд

Внутри бренда действия сгруппированы:

- `Работа с клиентами`
- `Работа с метриками`
- `Сотрудники`

`Работа с клиентами`:

1. `🟢 Выдать метрику`
2. `🟡 Списать метрику`
3. `🟢 Начислить монетки`
4. `🟡 Списать монетки`
5. `Балансы клиента`

`Работа с метриками`:

- `Все метрики`
- `Создать метрику`

Последнее изменение:

- При выборе бренда автопереход теперь обобщен.
- Код собирает список доступных разделов:
  - client work, если есть `CanIssue || CanRedeem || CanViewBalances`
  - metric work, если `CanManageMetrics`
  - staff, если `CanManageStaff`
- Если доступен ровно один раздел, бот сразу открывает его.
- Если доступно больше одного раздела, показывает экран бренда с кнопками.
- Это сделано в `src/StampService.TelegramBot/Features/Brands/Endpoints/BrandWorkspaceEndpoint.cs`.

Важный нюанс:

- Для единственного раздела `Сотрудники` текущая реализация напрямую открывает `BrandStaffListScreen`.
- Нужно убедиться, что контекст staff (`StaffSessionKeys.BrandId`, `StaffSessionKeys.BrandName`) корректен. Есть `StaffBrandContext`, который умеет fallback на `BrandWorkspaceScreen.BrandIdSessionKey` и `BrandWorkspaceScreen.BrandNameSessionKey`, поэтому это должно работать. Но build/test не запускались.

## 6. Локальная библиотека TelegramBotFlow

Библиотека лежит в `external/telegram-bot-flow`.

Что было исправлено:

- `BackButton`/`CloseButton` теперь по умолчанию `Назад`.
- `NavigationState.IsActionViewActive` добавлен как публичный getter/internal setter.
- `ScreenManager.ShowViewAsync` ставит `IsActionViewActive = true`.
- `ScreenManager.RenderScreenAsync` ставит `IsActionViewActive = false`.
- `NavigationService.NavigateBackAsync` сначала закрывает action-view через re-render текущего screen без pop из stack.
- `ScreenManager` теперь всегда делает `SetPending(view.PendingInputActionId)`, чтобы старые pending input не залипали.

Если пользователь попросит написать автору библиотеки:

- Проблема была в том, что `ShowView` для transient/action view не отличался от полноценной навигации по screen stack.
- Back после action view должен сначала закрывать/перерисовывать текущий screen, а не pop-ать предыдущий screen из history.
- Иначе `Back` часто возвращает в главное меню или перескакивает дальше ожидаемого состояния.

## 7. Ошибки и Application layer

Ранее был отдельный фокус на Errors/Application layer, потому что пользователь не доверял коду, написанному другим ИИ.

Что важно сохранить:

- Использовать `FluentResults`.
- Ошибки должны быть typed/structured, через `AppError`/`AppErrorCodes`, а не случайные строки.
- В Telegram слое форматировать ошибки через `BotErrorFormatter`.
- Не протаскивать infrastructure exceptions в UX.
- В Application layer не писать "лишь бы работало"; ошибки должны быть нормальной частью контракта use case.

## 10. Следующие бизнес-фичи, которые обсуждались как возможные

Пользователь хочет MVP и продажи, поэтому полезные следующие направления:

- Кампании/акции: временные правила начисления, например x2 монетки в определенные дни.
- Реферальные механики: пригласи друга, получи монетки.
- Простая CRM-карточка клиента: имя, CustomerCode, балансы, история, заметки.
- Сегменты клиентов: новые, активные, давно не были.
- Broadcast/рассылки по клиентам бренда, осторожно с Telegram constraints.
- Отчеты для владельца: сколько начислено/списано за период, топ клиентов, liabilities по монеткам.
- Срок жизни монеток/метрик, если бизнесу понадобится ограничивать обязательства.
- Импорт/экспорт клиентов.

Но ближайшее состояние проекта сейчас связано с завершением UX кошелька/истории и проверкой build/test после разрешения пользователя.
