# Операционный контекст StampService

Актуально на 2026-05-17, Europe/Moscow.

Этот файл нужен, чтобы новый чат быстро понял состояние проекта и мог сразу продолжать разработку без повторного восстановления архитектуры по коду. Это не changelog и не список последних фич. Здесь зафиксированы устойчивые договоренности, доменные правила, ключевые точки входа и ограничения.

## Рабочие договоренности

- Общение с пользователем вести на русском языке.
- Проект находится в `C:\Programmer\StampService\StampService`.
- Перед изменениями читать существующие паттерны проекта и двигаться в их стиле.
- Для поиска по коду сначала использовать `rg` и `rg --files`; они быстрее и обычно дают достаточно контекста.
- Не откатывать чужие изменения в рабочем дереве без явного разрешения пользователя.
- Ручные правки файлов делать через `apply_patch`.
- Русскую кодировку и уже существующие русские строки не нормализовать и не переписывать без необходимости.
- Build/test запускать только когда это нужно для задачи или пользователь просит. Если не запускались, честно писать об этом в финальном ответе.
- EF migrations не писать руками. Новые миграции создавать только через `dotnet ef migrations add`.
- Инфраструктурные предупреждения NuGet audit вроде `NU1900` при закрытой сети могут быть шумом, а не ошибкой кода.

## Назначение проекта

StampService - loyalty-сервис для брендов с основным пользовательским интерфейсом через Telegram-бота.

Ключевая идея: владелец бренда настраивает правила лояльности, сотрудники начисляют или списывают клиентские ресурсы, клиент видит в Telegram кошелек, баланс, доступные награды и историю. Система должна оставаться консистентной: балансы, транзакции, товары, метрики и роли меняются через доменные/Application-сценарии, а не случайными прямыми вставками в БД.

Текущий архитектурный путь - modular monolith. Микросервисы не делать без отдельного архитектурного решения.

## Структура решения

- `src/StampService.Domain` - доменные сущности, value objects, фабрики и бизнес-инварианты.
- `src/StampService.Application` - use cases, commands/queries, интерфейсы репозиториев, сервисы ledger, правила доступа.
- `src/StampService.Contracts` - DTO и контракты между слоями.
- `src/StampService.Infrastructure` - EF Core/PostgreSQL, конфигурации сущностей, репозитории, миграции, seeding/reset.
- `src/StampService.API` - HTTP API.
- `src/StampService.TelegramBot` - Telegram UX, экраны, endpoint actions, navigation, background services.
- `external/telegram-bot-flow` - локальная библиотека flow/navigation для Telegram-бота.
- `Tests/*` - доменные, application и API тесты.

Важные точки входа:

- `src/StampService.API/Program.cs` - bootstrap API и инфраструктурные режимы запуска.
- `src/StampService.API/Controllers/ApiControllerBase.cs` - authenticated `UserId` берется из JWT claims.
- `src/StampService.Application/DependencyInjection.cs` - регистрация Application-слоя, включая scanning command/query handlers.
- `src/StampService.TelegramBot/Common/Routing/BotEndpointHelpers.cs` - общие Telegram helper-методы.
- `external/telegram-bot-flow` - смотреть перед изменениями навигации, callbacks, `IBotAction`, `ScreenView`.

## Основная доменная модель

Пользователи и идентичность:

- `User` - пользователь системы.
- `UserIdentity` - внешняя идентичность пользователя, включая Telegram.
- `RedemptionCode` - одноразовый 4-значный код пользователя для операций сотрудника.
- `CustomerDigestState` - состояние digest-уведомлений клиента.

Бренды и доступы:

- `Brand` - бренд, в рамках которого живут настройки лояльности.
- `BrandMembership` - членство пользователя в бренде.
- `Role` - системная роль внутри бренда, базово `OWNER` и `STAFF`.

Метрики:

- `LoyaltyMetricDefinition` - определение метрики бренда.
- `MetricBalance` - материализованный баланс клиента по метрике.
- `StampTransaction` - ledger-запись изменения метрики.

Монеты и товары:

- `CoinWallet` - материализованный баланс монет клиента в бренде.
- `CoinTransaction` - ledger-запись изменения монет.
- `CoinProduct` - товар или награда за монеты.

Digest:

- `RewardDigestSettings` - редактируемые настройки системных digest-уведомлений.

## Бренды, владельцы и роли

- Бренд создается только вместе с владельцем через `CreateBrandWithOwnerCommand` / `CreateBrandWithOwnerHandler`.
- Смена владельца выполняется через `ReassignBrandOwnerCommand` / `ReassignBrandOwnerHandler`.
- Не возвращать старый сценарий создания бренда без владельца без явного запроса.
- Системный администратор определяется через `Admin:TelegramUserIds` в конфиге TelegramBot, а не через роли в БД.
- `RoleSeeder.SeedSystemRolesAsync` гарантирует наличие ролей `OWNER` и `STAFF`.
- Админ системы и владелец бренда - разные концепции. Админ может управлять системными инструментами, владелец управляет настройками своего бренда.

## Feature toggles бренда

У бренда есть независимые настройки:

- `IsMetricsEnabled`
- `IsCoinsEnabled`
- `IsCoinProductRedemptionEnabled`
- `IsManualCoinRedemptionEnabled`

Если метрики или монеты отключены, пользовательский UX не должен показывать соответствующие разделы, подсказки и действия. Настройки метрик, монет и способов списания управляются владельцем в рабочем пространстве бренда, а не глобальной админкой.

Для демонстрационных и тестовых данных разные бренды должны иметь разные комбинации этих настроек, чтобы можно было проверять поведение UI при включенных и выключенных возможностях.

## Ledger и консистентность

Баланс не должен быть первичным источником бизнес-операции. Операция должна создавать ledger-запись и синхронизировать материализованный баланс.

Основные сервисы:

- `ICoinLedgerService` / `CoinLedgerService` - создает или синхронизирует `CoinWallet`, пишет `CoinTransaction`, поддерживает баланс монет.
- `IMetricLedgerService` / `MetricLedgerService` - создает или синхронизирует `MetricBalance`, пишет `StampTransaction`, поддерживает баланс метрик.

Для начислений, списаний, покупок, истории и имитации данных использовать Application/domain сценарии и ledger-сервисы. Прямые insert/update в balances и transactions допустимы только для инфраструктурных задач вроде полного reset БД, если это осознанный технический сценарий.

## Метрики

- В UX используется пользовательское название метрики.
- Технический code метрики не должен показываться пользователю.
- `RedemptionAmount` определяет, сколько единиц метрики нужно для награды или списания.
- Реализованы сценарии списка, деталей, создания, редактирования, начисления, списания, истории и балансов.
- В истории клиента метрики и монеты должны быть разделены. Если соответствующая feature отключена у бренда, блок скрывается.

## Монеты и товары

Монеты - отдельный поддомен, не разновидность метрики.

Основные сценарии:

- начисление монет;
- ручное списание монет, если включено;
- покупка `CoinProduct`, если включено списание товаров;
- история операций;
- отображение доступных и недоступных товаров по балансу.

Покупка товара за монеты должна идти через подтверждение:

1. сотрудник вводит код пользователя;
2. бот показывает товары;
3. кнопки товаров показывают баланс и цену;
4. если баланса не хватает, списание не выполняется;
5. если хватает, сначала показывается подтверждение;
6. списание выполняется только после подтверждения;
7. отмена отменяет сценарий.

Комментарий transaction при покупке товара должен быть человеческим названием товара, без технических префиксов.

## Код пользователя

- `RedemptionCode` - одноразовый 4-значный код.
- Код живет 5 минут.
- Клиент показывает код сотруднику для операций начисления или списания.
- В пользовательском тексте писать "код пользователя", не `CustomerCode`.
- `CustomerCode` допустим только как техническое имя в коде.

## Telegram UX

Главное меню:

- `Мой кошелек`
- `Админка`, если Telegram id есть в `Admin:TelegramUserIds`
- если у пользователя один рабочий бренд - кнопка этого бренда;
- если рабочих брендов несколько - `Рабочие бренды`.

Правила UX:

- На самом главном меню не добавлять лишнюю кнопку `В главное меню`.
- В админских и вложенных экранах кнопка `В главное меню` должна быть самой нижней.
- Не плодить кнопки возврата в текущий или предыдущий подраздел. Если человеку нужно вернуться в раздел, он идет через главное меню.
- Не показывать технические поля вроде `ActorUserId`.
- Не показывать отключенные возможности бренда.
- Тексты должны быть пользовательскими и понятными, без внутренних слов в названиях клиентских сущностей.

Навигация:

- Для action/view navigation смотреть `external/telegram-bot-flow`.
- `IBotAction` находится в `TelegramBotFlow.Core.Endpoints`.
- `ScreenView.NavigateButton<TScreen>()` создает callback вида `nav:<screen_id>`.
- Для `MyWalletScreen` используется screen id `my_wallet`.

## Кошелек клиента

`MyWalletScreen` показывает:

- код пользователя;
- срок действия кода;
- балансы по брендам;
- доступные награды;
- историю и данные по включенным возможностям бренда;
- кнопку обновления кода.

При открытии кошелька обновляется `LastWalletOpenedAtUtc`, это важно для digest-логики.

В overview кошелька показывать только доступные товары и метрики. Недоступные пользователь может видеть при переходе внутрь конкретного бренда, если такой экран предусмотрен текущим UX.

## Reward Digest и уведомления

Reward Digest - системное уведомление клиента о доступных наградах. Это не брендовая рассылка.

Условия отправки:

- есть доступная награда;
- бренд и награда активны;
- баланс клиента достаточен;
- прошли интервалы после последнего digest и последнего открытия кошелька;
- настройки digest включены.

Состояние и настройки:

- `CustomerDigestState.LastDigestSentAtUtc`
- `CustomerDigestState.LastWalletOpenedAtUtc`
- `RewardDigestSettings.Enabled`
- `RewardDigestSettings.MessageToUserIntervalMinutes`
- `RewardDigestSettings.ScanIntervalMinutes`
- `RewardDigestSettings.BatchSize`
- `RewardDigestSettings.MaxBrandsPerMessage`
- `RewardDigestSettings.MaxRewardsPerBrand`

`RewardDigestSettings` редактируются через админку без перезапуска. `RewardDigestOptions` используется как fallback для первого создания настроек.

Отправка:

- `CustomerRewardDigestHostedService` периодически сканирует пользователей и перечитывает настройки.
- `CustomerNotificationService` и `CustomerRewardDigestSender` отправляют direct notifications через Telegram.
- После отправки notification нужно сбрасывать nav message, чтобы следующий экран открывался новым сообщением.

Outbox сейчас не вводить без отдельного решения. Текущая модель уведомлений - прямая отправка через Telegram.

## Админка и демо-инструменты

Глобальная админка доступна только системному администратору из `Admin:TelegramUserIds`.

В админке есть системные инструменты:

- управление брендами и владельцами;
- настройки digest;
- раздел `Работа с демо` для демонстрационных данных;
- полный reset БД с подтверждением.

Демо-инструменты - часть системы для быстрой подготовки стенда и демонстрации клиентского UX. Они не должны нарушать доменные правила.

Сценарии:

- создание набора заранее настроенных брендов для текущего системного администратора как владельца;
- создание данных указанному пользователю по его 4-значному коду в выбранном бренде владельца;
- очистка всей БД только с отдельным экраном подтверждения и кнопкой `Да, очистить`;
- после очистки БД бот должен сбросить текущую навигацию и сессию, потому что данные текущих экранов могли исчезнуть.

Демонстрационные данные должны выглядеть как настоящие:

- в пользовательских названиях брендов, метрик, товаров и комментариев не использовать слова вроде "демо", "тест", "монетки", "скрытый" и другие внутренние обозначения;
- создавать метрики, товары и настройки через доменные фабрики и Application-сценарии;
- генерировать такую историю, где часть наград доступна, а на часть ресурсов не хватает;
- не создавать лишних пользователей, если задача требует работать только с системным администратором или конкретно указанным клиентом.

Ключевые файлы:

- `src/StampService.Application/Demo`
- `src/StampService.Application/Demo/DemoDataCatalog.cs` - каталог реалистичных шаблонов брендов, метрик, товаров и комментариев.
- `src/StampService.Infrastructure/Seeding`
- `src/StampService.Infrastructure/Seeding/DemoDataSeeder.cs` - инфраструктурный seed для подготовки стенда.
- `src/StampService.TelegramBot` screens/actions админского demo-раздела

Также есть CLI-сценарий `--seed-demo` в `src/StampService.API/Program.cs`. Он нужен для инфраструктурного заполнения стенда и должен учитывать системного администратора из `Admin:TelegramUserIds`.

## База данных и инфраструктура

Основная БД - PostgreSQL через EF Core.

Инфраструктурные операции:

- миграции - только через EF tooling;
- полный reset для demo/admin tooling может использовать инфраструктурный `TRUNCATE ... RESTART IDENTITY CASCADE`, после чего нужно восстановить системные роли;
- seeding должен учитывать `Admin:TelegramUserIds`, потому что системный администратор не является обычной ролью в БД.

При изменениях БД:

- проверить доменные инварианты;
- обновить EF configuration;
- создать миграцию через `dotnet ef migrations add`;
- не править migration вручную, кроме явно осознанной минимальной корректировки после генерации, если это действительно необходимо.

## Что не делать без явного запроса

- Не переводить проект на микросервисы.
- Не вводить outbox вместо текущих direct Telegram notifications.
- Не менять модель системного администратора на роль в БД.
- Не возвращать создание бренда без владельца.
- Не показывать пользователю технические коды, ids и служебные поля.
- Не добавлять лишние кнопки навигации в Telegram UI.
- Не делать демо-данные прямыми вставками в business tables в обход доменных правил.
- Не переписывать русские строки только ради косметики или нормализации кодировки.

## Как подходить к новым задачам

1. Сначала определить слой, которому принадлежит изменение: Domain, Application, Infrastructure, API или TelegramBot.
2. Найти аналогичный существующий сценарий и повторить его стиль.
3. Если изменение касается балансов, транзакций, метрик, монет или товаров, использовать ledger/domain/Application-путь.
4. Если изменение касается Telegram UX, проверить navigation helpers и существующие screen/action patterns.
5. Если изменение касается доступов, различать системного администратора, владельца бренда и сотрудника.
6. После реализации проверить, что отключенные feature toggles не протекают в UX.
7. В финальном ответе кратко указать, что изменено и какие проверки запускались.
