# React Web UI правила

Актуально на 2026-06-17. Web развивается в `src/StampService.Web` как React/TypeScript app поверх typed API clients.

## Общая модель

- Web UI - полноценное React-приложение, не static `wwwroot`.
- Старые `/app`, `/phone-auth-test`, `/design-variants` не являются целевыми web surfaces и не должны восстанавливаться без отдельной причины.
- UI тексты на русском, пользовательские, без technical ids/debug/internal codes.
- Пользовательский термин для metrics в web - `Штампы`; внутренние `Metric*`, `metrics`, API endpoints не переименовывать без отдельного решения.
- Не переносить бизнес-правила во frontend. Frontend собирает ввод, вызывает typed clients, показывает результат и UX-состояния.

## Auth UI

- Phone login через OTP.
- На экране входа два действия:
  - `Получить код`: `sendSms=false`, OTP администратору в Telegram;
  - `Получить код по SMS`: `sendSms=true`, только если публичная SMS-настройка включена.
- Frontend проверяет SMS setting для UX, но backend остается enforcement.
- JWT хранится в отдельном auth/API слое; не размазывать работу с токеном по компонентам.
- `401` обрабатывать централизованно.
- Logout находится в разделе `Аккаунт`, не в общем header.

## Телефонный ввод

- Пользовательский ввод нормализуется к `+79991234567`.
- Базовая валидация соответствует E.164; для `+7` требуется 11 цифр вместе с кодом страны.
- Web phone input использует российскую маску `+7 (999) 123-45-67`, но backend остается источником истины.
- Soft-deleted phone identity - история, не допустимый способ входа или поиска.

## Навигация и роли

- Основная навигация строится из одного источника (`navigationLabels.ts` + access state), без отдельной mobile role matrix.
- `Кошелёк`, `Работа`/`Бренды`, `Админка`, `Аккаунт` - основные разделы.
- `Админка` показывается отдельным пунктом, если `GET /api/admin/access` разрешил доступ.
- Повторный клик по `Кошелёк` или `Работа`/`Бренды` - home/root action раздела, а не noop.
- Локальные кнопки `Назад` в wallet/brand workspace не возвращать точечно; если нужен новый возврат, проектировать его как часть общей навигации.

## Mobile layout

- Основная адаптация в `src/StampService.Web/src/styles.css`.
- `max-width: 760px` - основной mobile breakpoint с bottom navigation.
- `max-width: 420px` - дополнительный узкий breakpoint.
- Desktop grid-классы должны иметь mobile override.
- Длинные тексты, статусы, кнопки и коды списания не должны расширять viewport: использовать `min-width: 0`, переносы, полноширинные действия.
- Новые экраны сразу проектировать для mobile; не рассчитывать на `overflow-x` как решение.

## Wallet UI

- `WalletPage` загружает overview, код списания, список брендов и переход в бренд.
- Карточка бренда в `Мои награды` кликабельна целиком; отдельная кнопка `Открыть` внутри не нужна.
- `BrandWalletPage` показывает brand hero, `RedemptionCodeCard`, затем `WalletBrandDetailsBlock`.
- `WalletBrandDetailsBlock` - общий блок вкладок `Штампы` / `Монетки` / `История`, переиспользуется и в кошельке, и в карточке клиента workspace.
- Вкладки строятся по DTO/details: `Штампы` только если включены metrics, `Монетки` только если включены coins, `История` всегда общая.

## Brand workspace

Workspace - mobile-first рабочая консоль сотрудника/владельца.

- Первое состояние: поиск клиента по телефону.
- Поиск не создает клиента и не добавляет его в бренд.
- `found=false` открывает локальную draft-карточку нового клиента; это не backend-сущность.
- `Недавние номера` - локальный `localStorage` cache до 6 успешно открытых телефонов по `brandId`; не бизнес-аудит и не source of truth.
- Второе состояние: выбранный клиент. Обычная карточка показывает имя, телефон и `WalletBrandDetailsBlock`.
- Draft-карточка показывает `Новый клиент`, телефон и явное создание; блок операций не показывается.
- После `createBrandCustomerByPhone` обязательно повторно вызвать `getBrandCustomerCard` и заменить draft на обычную карточку.
- Issue-by-phone endpoints не использовать как скрытое создание клиента из draft.
- Операции над выбранным клиентом используют телефон из карточки.
- Списание штампов, списание монеток и выдача товара за монетки требуют одноразовый код списания клиента.
- UI-сортировка options допустима только как presentation: доступные награды выше недоступных.

Компонентные границы:

- `BrandWorkspacePage` - контейнер: selected workspace, `getBrandWorkspace`, selector/workspace.
- `BrandSelector` - список рабочих брендов.
- `BrandWorkspace` - рабочая область выбранного бренда: поиск, клиент, настройки.
- `BrandCustomerSearchScreen` - поиск телефона, `getBrandCustomerCard`, not found/error, недавние номера.
- `SelectedCustomerWorkspace` внутри `BrandWorkspace` - экран выбранного клиента.
- `BrandSettingsPage` внутри `BrandWorkspace` - отдельный экран управления брендом.

## Brand settings и admin

- Управление брендом открывается кнопкой-шестеренкой в плашке бренда, если есть permissions.
- В brand settings доступны `Штампы`, `Товары`, `Сотрудники`, `Бренд` по permissions.
- Welcome rewards настраиваются в разделе бренда и должны учитывать feature toggles.
- Недоступные разделы не показывать, а не disabled.
- Глобальная админка содержит системные настройки SMS-кодов; бинарная настройка отображается тумблером, не кнопками включить/выключить.

## Визуальные договоренности

- Web-навигация без emoji.
- Дизайн спокойный, рабочий, пригодный для клиента, сотрудника и владельца.
- Карточками являются формы, options, результаты, модальные/bottom sheet панели; не оборачивать весь workspace в большую вложенную surface без необходимости.
- Верх workspace: плашка бренда с названием, человекочитаемой ролью и шестеренкой; flags/chips/technical fields там не показывать.
- Segment controls использовать для выбора режимов/уровней.
