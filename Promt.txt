# ГЛАВНЫЕ ПРИНЦИПЫ РАЗРАБОТКИ (СТРОГО СОБЛЮДАТЬ)
1. **YAGNI (No Over-engineering):** Не предлагать поля, методы или сущности "на будущее". Реализуем только то, что требуется для текущего функционала MVP.
2. **Domain-Driven Design (DDD):** 
   - Сущности должны инкапсулировать логику (методы изменения состояния внутри класса).
   - Использовать Value Objects там, где это упрощает логику (но без фанатизма).
   - Запретить прямое изменение свойств извне (использовать `private set` / `protected set` и методы-команды).
3. **Multi-tenancy через Brand:** 
   - Основная единица изоляции — `Brand` (Сеть/Бренд).
   - `Location` (Точка/Филиал) принадлежит `Brand`.
   - Пользователи (`User`) взаимодействуют с конкретным `Brand`.
4. **Гибкая система метрик:** 
   - Бренды сами определяют, что начислять (кофе, стрижки, пироги). Нет жесткого Enum в коде.
   - Используем сущность `LoyaltyMetricDefinition` для определения типов начислений.
5. **Soft Delete:** Все бизнес-сущности наследуются от `BaseEntity` (имеют `DeletedAt`). Технические справочники (`Role`, `LoyaltyMetricDefinition`) могут не иметь Soft Delete, если не планируется их удаление.

# ТЕКУЩАЯ АРХИТЕКТУРА И СУЩНОСТИ

## 1. Фундамент (`StampService.Domain.Shared`)
- `ISoftDelete`: интерфейс с `DateTime? DeletedAt`.
- `BaseEntity`: абстрактный класс, реализует `ISoftDelete`, содержит `Guid Id`, `DateTime CreatedAt`, `DateTime? UpdatedAt`, `DateTime? DeletedAt`.
  - Метод `Touch()`: обновляет `UpdatedAt` (вызывается при изменении сущности, НЕ при создании).
  - **Note:** `UpdatedAt` — nullable, при создании `null`, проставляется только через `Touch()`.

## 2. Структура Бизнеса (`StampService.Domain.Brand`)
- **`Brand`**: Агрегат верхнего уровня.
  - Поля: `Name`, `Slug`, `Settings` (JSON).
  - Коллекция: `_locations` (HashSet<Location>).
  - Метод: `AddLocation(...)`.
- **`Location`**: Филиал бренда.
  - Поля: `BrandId`, `Name` (Value Object), `Address` (Value Object), `IsActive`.
  - Связь: Многие к одному с `Brand`.
- **Value Objects**: `LocationName`, `Address` (immutable records/classes с валидацией).

## 3. Пользователи и Доступ (`StampService.Domain.User` & `Access`)
- **`User`**: Личность.
  - Поля: `Name`, `AvatarUrl`.
  - Коллекция: `UserIdentities`.
- **`UserIdentity`**: Способ входа (Telegram, Email).
  - Поля: `UserId`, `Provider` (Enum), `ProviderKey`, `Metadata` (JSON).
- **`Role`**: Системные роли (`OWNER`, `STAFF`, `CUSTOMER`).
  - Поля: `SystemName`, `DisplayName`. (Без Soft Delete).
- **`BrandMembership`**: Связь User <-> Brand <-> Role.
  - Поля: `UserId`, `BrandId`, `RoleId`, `LocationId` (nullable, для привязки к точке).
  - Soft Delete: Да (для восстановления доступа).

## 4. Лояльность (`StampService.Domain.Loyalty`)
- **`LoyaltyMetricDefinition`**: Определение того, ЧТО начисляем (конфигурация бренда).
  - Поля: `BrandId`, `Code` (уникальный код, напр. "COFFEE"), `Name` (напр. "Чашка кофе"), `IsActive`.
  - Методы: `Create()`, `Deactivate()`, `Activate()`, `UpdateName()`, `UpdateCode()`.
  - Логика: Бренд сам создает нужные ему метрики.
  - ✅ **Реализовано**
- **`MetricBalance`**: Текущий баланс пользователя по конкретной метрике.
  - Поля: `UserId`, `BrandId`, `MetricDefinitionId`, `Value` (int).
  - Навигационные свойства: `User` (UserEntity), `Brand` (BrandEntity), `MetricDefinition`.
  - Уникальность: `UserId + BrandId + MetricDefinitionId`.
  - Методы: `Add(int)` / `Subtract(int)` (с проверкой на отрицательный баланс). `StampTransaction` создаётся в Application слое.
  - **Note:** Используется `using alias` для namespace конфликта (`UserEntity` = `StampService.Domain.User.User`, `BrandEntity` = `StampService.Domain.Brand.Brand`).
  - ✅ **Реализовано**
- **`StampTransaction`**: История операций с балансом.
  - Поля: `AccountBalanceId` (FK к MetricBalance), `MetricDefinitionId` (дублируется для быстрого поиска), `Amount` (int, может быть отрицательным), `Comment`.
  - Создаётся в Application-слое при каждом изменении баланса.
  - ✅ **Реализовано**
